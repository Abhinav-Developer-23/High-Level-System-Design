# Scalability — Question Bank

## Beginner Level
1. What is scalability in the context of system design?

<details><summary>Answer</summary>Scalability is the ability of a system to handle increased load by adding resources — without requiring a complete architectural overhaul. The key word is "ability": a scalable system can grow to meet demand gracefully. For example, a system that serves 1,000 users today should be able to serve 100,000 users by adding more servers, upgrading hardware, or optimizing its architecture, rather than being rewritten from scratch.</details>

2. What is the difference between vertical scaling and horizontal scaling?

<details><summary>Answer</summary>**Vertical scaling (scale up)** means adding more power to an existing machine — more CPU, RAM, faster SSDs, or better network cards. You upgrade the server itself. **Horizontal scaling (scale out)** means adding more machines to distribute the load. Instead of one powerful server, you use many commodity servers working together behind a load balancer. Vertical scaling is simpler but has hardware limits; horizontal scaling is more complex but can grow indefinitely.</details>

3. What are the common metrics used to measure scalability?

<details><summary>Answer</summary>Scalability is measured along several dimensions: (1) **Requests per second (RPS)** — how many API calls the system handles. (2) **Concurrent users** — users active at the same time. (3) **Data volume** — amount of data stored or processed. (4) **Throughput** — data transferred per unit time (e.g., 1 GB/s). (5) **Query rate (QPS)** — database queries per second. (6) **Message rate** — messages processed through queues per second. These metrics help you quantify load and track whether the system can handle growth.</details>

4. What does "performance under load" mean in the context of scalability?

<details><summary>Answer</summary>Performance under load measures how well a system maintains acceptable response times as the number of requests or users increases. A well-scaling system shows linear or sublinear degradation — doubling the load does not double response time. For example, if baseline response time is 50ms at 1x load, a well-scaling system might show 55ms at 2x, 70ms at 5x, and 150ms at 10x. When response times spike exponentially or the system starts timing out, you've hit a scalability wall.</details>

5. What are the pros of vertical scaling?

<details><summary>Answer</summary>(1) **Simple** — no code changes required, just move to a bigger machine. (2) **Lower latency** — all data is local with no network hops between components. (3) **No distributed complexity** — a single server means no network partitions, no data synchronization issues, no split-brain scenarios. (4) **Easier debugging** — everything runs in one place, so logs, metrics, and traces are all local. Vertical scaling is often the right first step because of its simplicity.</details>

6. What are the cons of vertical scaling?

<details><summary>Answer</summary>(1) **Hardware limits** — you cannot scale beyond the largest available machine, even cloud providers have limits. (2) **Single point of failure** — one server means one failure point; if it goes down, everything goes down. (3) **Cost curve** — larger machines cost disproportionately more; doubling capacity often more than doubles cost. (4) **Downtime during upgrades** — migrating to a bigger machine typically requires downtime. These limitations are why most high-traffic systems eventually move to horizontal scaling.</details>

7. What are the pros of horizontal scaling?

<details><summary>Answer</summary>(1) **No hard limit** — you can keep adding servers as needed; cloud providers make this nearly unlimited. (2) **Fault tolerance** — if one server fails, others continue serving traffic; no single point of failure. (3) **Cost-effective** — many smaller commodity machines often cost less than one giant machine. (4) **Geographic distribution** — you can place servers closer to users in different regions for lower latency. Horizontal scaling is how companies like Google, Netflix, and Amazon handle billions of requests.</details>

8. What are the cons of horizontal scaling?

<details><summary>Answer</summary>(1) **Complexity** — distributed systems are harder to build, debug, and maintain. (2) **Data consistency** — keeping data synchronized across multiple servers is challenging. (3) **Network overhead** — communication between servers adds latency. (4) **Stateless requirement** — application servers typically need to be stateless, which may require architectural changes like moving session data to a shared store. These trade-offs mean horizontal scaling should be adopted when vertical scaling is no longer sufficient, not as a premature first step.</details>

9. What does it mean for a service to be "stateless"?

<details><summary>Answer</summary>A stateless service does not store any session or user-specific data locally on the server. Each incoming request can be handled by any server because no request depends on data from a previous request being on a specific server. Session data, if needed, is stored in a shared external store like Redis or Memcached. This makes horizontal scaling trivial — the load balancer can route any request to any server without worrying about affinity.</details>

10. Why is statelessness important for horizontal scaling?

<details><summary>Answer</summary>If a server stores session data locally (stateful), then all requests from a particular user must go to that same server ("sticky sessions"). This creates hotspots, makes it risky to remove or restart servers (users lose their sessions), and prevents the load balancer from freely distributing traffic. With stateless servers, the load balancer has complete freedom to distribute requests to any server, enabling easy auto-scaling and graceful handling of server failures.</details>

11. How do you make a service stateless?

<details><summary>Answer</summary>(1) **Store session data in a shared cache** like Redis or Memcached — any server can read the session. (2) **Use tokens (JWT)** instead of server-side sessions — the client carries the session state in a signed token. (3) **Store uploaded files in object storage** (e.g., S3) instead of local disk. (4) **Externalize all state** — any data that a request handler needs between requests should be in a database, cache, or message queue, not in local memory or disk.</details>

12. When is vertical scaling the right choice?

<details><summary>Answer</summary>Vertical scaling works well for: (1) **Databases where data locality matters** — before sharding becomes necessary, a single powerful database server avoids distributed query complexity. (2) **Applications with strong consistency requirements** — a single server eliminates coordination overhead. (3) **Early-stage startups** that need simplicity over scale. (4) **Workloads with predictable, moderate growth** — if you know the ceiling is manageable. Many real-world systems run on vertically scaled databases for years. The key is knowing when horizontal scaling becomes necessary.</details>

## Intermediate Level
13. What is a load balancer and why is it essential for horizontal scaling?

<details><summary>Answer</summary>A load balancer is a component that distributes incoming network traffic across multiple servers. It is essential for horizontal scaling because it: (1) evenly distributes requests to prevent any single server from being overwhelmed, (2) monitors server health and stops routing to failed servers, (3) enables adding or removing servers without downtime, and (4) can terminate TLS, offloading encryption work from application servers. Without a load balancer, clients would need to know about individual servers, making scaling and failover manual and error-prone.</details>

14. What is the difference between scaling the application tier vs the database tier?

<details><summary>Answer</summary>The **application tier** is usually the easiest to scale horizontally — you make servers stateless, put a load balancer in front, and add more instances. The **database tier** is typically the hardest to scale because it manages state. You cannot simply spin up more database instances and put a load balancer in front of them — data consistency, durability, and transaction isolation all complicate matters. Database scaling requires specialized strategies like read replicas, sharding, or moving to distributed databases.</details>

15. What are read replicas and when should you use them?

<details><summary>Answer</summary>Read replicas are copies of your primary database that serve read queries. The primary handles all writes, and changes are replicated to the replicas asynchronously. Use them when: (1) your read-to-write ratio is 10:1 or higher, (2) reads are the bottleneck, not writes, (3) you can tolerate slight replication lag (stale reads). **Pros**: simple to set up, offloads read traffic, provides read availability. **Cons**: doesn't help with write-heavy workloads, introduces replication lag, replicas consume full storage. Most applications are read-heavy, so this is often the first database scaling strategy.</details>

16. What is database sharding and why is it needed?

<details><summary>Answer</summary>Sharding (partitioning) splits your data across multiple database instances based on a partition key (e.g., user ID). Each shard holds a subset of the data. It's needed when: (1) read replicas aren't enough, (2) write volume exceeds what a single primary can handle, or (3) the dataset is too large for one machine. Sharding distributes both reads AND writes. However, it's complex — cross-shard queries are expensive, rebalancing shards is operationally difficult, and transactions across shards are very hard to implement.</details>

17. What are the common sharding strategies?

<details><summary>Answer</summary>(1) **Range-based sharding** — shard by value ranges (e.g., users A-H on shard 1, I-P on shard 2). Simple but can lead to uneven distribution if data is not uniformly distributed. (2) **Hash-based sharding** — hash the key and mod by number of shards. Provides more uniform distribution but makes range queries difficult. (3) **Directory-based sharding** — maintain a lookup table mapping keys to shards. Most flexible but the directory itself becomes a single point of failure and a potential bottleneck.</details>

18. How does caching help scalability?

<details><summary>Answer</summary>Caching reduces load on databases and improves response times by storing frequently accessed data in memory. A well-designed cache like Redis can handle 100,000+ operations per second on a single node — orders of magnitude more than a database. With a good caching strategy, 80–90% of reads never hit the database. This effectively multiplies your system's read capacity without adding database hardware. Caching is essential for high-traffic systems and is one of the highest-ROI scalability optimizations.</details>

19. What is the cache-aside pattern?

<details><summary>Answer</summary>In the cache-aside pattern, the application checks the cache first. On a cache hit, data is returned directly. On a cache miss, the application fetches data from the database, writes it into the cache, and then returns it. This pattern is simple and widely used. The application controls what goes into the cache and for how long (TTL). Trade-offs: (1) first request for any data always hits the database (cold start), (2) data can become stale if the database is updated without invalidating the cache, (3) cache stampede can occur if many requests miss the cache simultaneously.</details>

20. What is consistent hashing and how does it help scale caches?

<details><summary>Answer</summary>Consistent hashing distributes keys across cache nodes using a hash ring. When nodes are added or removed, only a small fraction of keys (`k/n`) need to be remapped, unlike modulo-based hashing where nearly all keys are remapped. This is critical for scaling caches because: (1) adding a cache node doesn't invalidate most cached data, (2) removing a failed node only affects the keys on that node, (3) it enables elastic scaling without massive cache miss storms. Redis Cluster uses hash slots (a form of consistent hashing) for automatic partitioning.</details>

21. How do message queues help with scalability?

<details><summary>Answer</summary>Message queues (e.g., Kafka, RabbitMQ) enable systems to scale by: (1) **Decoupling producers and consumers** — each can scale independently. (2) **Buffering traffic spikes** — the queue absorbs bursts so consumers can process at their own pace. (3) **Enabling parallel processing** — multiple consumers can process messages from the same topic concurrently. (4) **Preventing cascading failures** — if a downstream service is slow, messages queue up instead of blocking the producer. Kafka partitions allow parallel consumption, and adding more partitions increases throughput linearly.</details>

22. Explain the difference between sticky sessions and shared session stores.

<details><summary>Answer</summary>**Sticky sessions** (session affinity) tie a user to a specific server — the load balancer always routes that user's requests to the same server. This makes the server stateful: if it crashes, the user loses their session. It also creates hotspots and prevents free load distribution. **Shared session stores** (e.g., Redis, Memcached) store session data externally. Any server can read the session, making servers stateless. The load balancer can route freely, servers can be added/removed without session loss, and auto-scaling works seamlessly. Shared stores are the preferred approach for scalable systems.</details>

23. What is auto-scaling and how does it work?

<details><summary>Answer</summary>Auto-scaling automatically adjusts the number of running server instances based on current demand. You define scaling policies based on metrics like CPU utilization, memory usage, request count, or queue depth. When load exceeds a threshold (e.g., CPU > 70%), new instances are launched. When load drops, excess instances are terminated. Auto-scaling requires stateless services (so any instance can handle any request), health checks, and a load balancer. Cloud providers like AWS (Auto Scaling Groups), GCP, and Azure provide built-in auto-scaling capabilities.</details>

24. What is the role of a CDN in scaling a web application?

<details><summary>Answer</summary>A CDN (Content Delivery Network) scales the delivery of static assets (images, CSS, JS, videos) by serving them from edge servers distributed globally, close to users. This: (1) reduces latency — users fetch content from a nearby edge instead of the origin server. (2) reduces origin server load — the origin only serves cache misses. (3) absorbs traffic spikes — CDN edge nodes can handle massive concurrent requests. (4) provides bandwidth savings — less data flows from your origin. For read-heavy, content-heavy applications, a CDN is one of the most effective scaling tools.</details>

25. How does deploying across multiple availability zones improve scalability and reliability?

<details><summary>Answer</summary>Deploying across multiple availability zones (AZs) means running copies of your application and data in physically separate data centers within a region. This improves: (1) **Fault tolerance** — if one AZ goes down (power failure, network issue), the others continue serving traffic. (2) **Scalability** — you can distribute load across AZs, effectively multiplying capacity. (3) **Latency** — users are routed to the nearest AZ. The load balancer distributes traffic across AZs and reroutes on failure. Databases need multi-AZ replication for data durability.</details>

26. What is the difference between SQL and NoSQL databases in terms of scalability?

<details><summary>Answer</summary>**SQL databases** (PostgreSQL, MySQL) are designed for strong consistency and ACID transactions. They scale vertically well but are harder to scale horizontally — sharding requires significant effort and cross-shard queries/joins are expensive. **NoSQL databases** (Cassandra, MongoDB, DynamoDB) are designed for horizontal scaling from the ground up — they feature built-in sharding, automatic data distribution, and flexible schema. The trade-off is that NoSQL often uses eventual consistency, doesn't support joins (requiring denormalization), and has different query patterns. Choose based on your consistency and query requirements.</details>

27. What does "identify the bottleneck first" mean in the context of scaling?

<details><summary>Answer</summary>Before deciding how to scale, you must determine which component is the constraint. Adding more app servers doesn't help if the database is the problem. Adding read replicas doesn't help if writes are the bottleneck. The bottleneck is the slowest or most resource-constrained component — it could be CPU, memory, disk I/O, database, network, or an external dependency. Use monitoring and profiling to find it: high CPU on the DB server? Database is the bottleneck. App servers at 100% CPU? Scale the app tier. Always diagnose before prescribing a solution.</details>

28. How does database connection pooling help with scalability?

<details><summary>Answer</summary>Every database connection consumes memory and resources on the database server. Without pooling, each request creates a new connection (TCP handshake + auth), uses it, and closes it — wasting time and resources. A connection pool maintains a set of reusable connections. Requests borrow a connection, execute queries, and return it. This: (1) reduces database connection overhead (no repeated handshakes). (2) limits the total number of connections to the database (preventing exhaustion). (3) reduces per-request latency. Tools like PgBouncer (PostgreSQL) and HikariCP (Java) provide high-performance connection pooling.</details>

## Advanced Level
29. Explain the concept of linear vs sublinear vs superlinear scaling.

<details><summary>Answer</summary>**Linear scaling** means performance degrades proportionally to load — doubling load roughly doubles response time. This is predictable and manageable. **Sublinear scaling** means performance degrades less than proportionally — doubling load increases response time by less than 2x. This happens when caching is effective or fixed costs are amortized. **Superlinear scaling** (bad) means performance degrades faster than proportionally — doubling load more than doubles response time. This indicates a bottleneck: contention, lock congestion, or resource exhaustion. The goal is linear or sublinear scaling; superlinear scaling signals an architectural problem that must be addressed.</details>

30. What is a scalability ceiling and how do you identify one?

<details><summary>Answer</summary>A scalability ceiling is the point at which adding more resources no longer improves performance (or makes it worse). Symptoms include: response times spiking despite adding servers, throughput plateauing, error rates climbing, and resource utilization becoming uneven. Common causes: (1) database is a single bottleneck — adding app servers doesn't help. (2) lock contention — more threads causes more waiting. (3) single-threaded components in the path. (4) network bandwidth saturation. (5) Amdahl's Law — the serial portion limits overall speedup. Identify ceilings through load testing and monitoring component-level metrics.</details>

31. How does sharding affect cross-shard queries and transactions?

<details><summary>Answer</summary>Sharding distributes data across multiple databases, but some operations need data from multiple shards. **Cross-shard queries** (e.g., "find all orders > $100 across all users") require querying every shard and merging results — this is slow and expensive, sometimes called "scatter-gather." **Cross-shard transactions** are even harder — maintaining ACID properties across independent databases requires distributed transaction protocols like two-phase commit (2PC), which are slow and reduce availability. Best practice: design your shard key so that frequently accessed data co-locates on the same shard, minimizing cross-shard operations.</details>

32. What is a shard key and what makes a good one?

<details><summary>Answer</summary>A shard key is the field used to determine which shard stores a piece of data. A good shard key has: (1) **High cardinality** — many unique values to distribute data evenly. (2) **Even distribution** — avoids hotspots where one shard gets disproportionately more data. (3) **Query alignment** — most queries filter on the shard key, avoiding cross-shard lookups. (4) **Stability** — the key doesn't change frequently (changing requires data migration). Example: `user_id` is often a good shard key for user-centric applications because it distributes evenly and most queries are per-user. `country` is usually bad — uneven distribution (most users in a few countries).</details>

33. What is the "thundering herd" or "cache stampede" problem and how does it relate to scalability?

<details><summary>Answer</summary>A cache stampede occurs when a popular cache entry expires and many concurrent requests simultaneously miss the cache and hit the database. If 10,000 requests/second are served from cache and the key expires, all 10,000 suddenly hit the database, potentially overwhelming it. Solutions: (1) **Lock/mutex** — only one request fetches from the database; others wait for the cache to be repopulated. (2) **Probabilistic early expiration** — randomly refresh before actual expiry. (3) **Background refresh** — a background job repopulates the cache before it expires. (4) **Stale-while-revalidate** — serve stale data while fetching fresh data in the background.</details>

34. How does Amdahl's Law apply to scaling distributed systems?

<details><summary>Answer</summary>Amdahl's Law states that the maximum speedup from parallelization is limited by the serial (non-parallelizable) portion of the workload. If 10% of your request processing is inherently serial (e.g., acquiring a global lock, writing to a single primary), then maximum speedup is 10x regardless of how many servers you add. In distributed systems, serial bottlenecks include: single primary database writes, global coordination (distributed locks, leader election), and sequential processing steps. To scale beyond these limits, you must identify and eliminate serial sections — e.g., shard the database to remove the single-writer bottleneck.</details>

35. What is a circuit breaker pattern and how does it help scalability?

<details><summary>Answer</summary>A circuit breaker prevents cascading failures in distributed systems. When a downstream service is failing or slow, the circuit breaker "opens" and immediately returns an error (or a fallback response) without making the downstream call. After a cooldown period, it allows a few "probe" requests through — if they succeed, the circuit "closes" and normal traffic resumes. This helps scalability by: (1) preventing thread/connection pool exhaustion from slow downstream calls, (2) reducing load on the already-struggling downstream service, (3) allowing the rest of the system to continue functioning. Libraries: Hystrix, Resilience4j, Polly.</details>

36. How does rate limiting help protect a system's scalability?

<details><summary>Answer</summary>Rate limiting restricts the number of requests a client can make within a time window (e.g., 100 requests/minute per API key). It protects scalability by: (1) preventing any single client from consuming disproportionate resources, (2) protecting against DDoS attacks and abusive traffic patterns, (3) ensuring fair usage across all clients, (4) preventing cascading failures from traffic spikes. Common algorithms: token bucket, sliding window, fixed window. Rate limits should be applied at the API gateway or load balancer level. Without rate limiting, a single misbehaving client can degrade performance for everyone.</details>

37. What is backpressure and how does it relate to scalability?

<details><summary>Answer</summary>Backpressure is a mechanism where a slow consumer signals to a fast producer to slow down, preventing buffer overflow and resource exhaustion. Without backpressure, a fast producer overwhelms a slow consumer — queues grow unbounded, memory fills up, and the system crashes. In scalable systems, backpressure appears in: (1) message queues — consumers process at their own pace, the queue buffers the difference. (2) reactive streams — frameworks like RxJava and Reactor support backpressure natively. (3) TCP flow control — the receiver advertises a window size to control the sender's rate. Backpressure is essential for maintaining system stability under varying load.</details>

38. How do you handle hot partitions or hot keys in a sharded system?

<details><summary>Answer</summary>A hot partition occurs when one shard receives disproportionately more traffic — e.g., a celebrity's profile in a social media app sharded by user ID. Solutions: (1) **Key splitting** — append a random suffix to the hot key and distribute across multiple shards (e.g., `celebrity_123_0`, `celebrity_123_1`, ...`celebrity_123_9`), aggregating reads. (2) **Dedicated shard** — move the hot key to its own shard on more powerful hardware. (3) **Caching** — aggressively cache hot keys to avoid hitting the shard. (4) **Salting** — add a salt to the shard key for better distribution. (5) **Redesign** — choose a different shard key that distributes the problematic data more evenly.</details>

39. Explain the difference between scaling reads vs scaling writes.

<details><summary>Answer</summary>**Scaling reads** is relatively straightforward: add read replicas, add caching layers (Redis, CDN), use denormalized views, or add more app servers. Reads don't modify data, so consistency is the main concern (replication lag). **Scaling writes** is much harder: every write must eventually reach the source of truth and be durable. Strategies include: sharding (distribute writes across multiple primaries), write-behind caching (batch writes), message queues (buffer and batch writes), multi-master replication (with conflict resolution), and choosing eventual consistency to reduce coordination overhead. Most systems are read-heavy, which is why read scaling strategies are more commonly discussed.</details>

40. What is the "shared-nothing" architecture and why is it important for scalability?

<details><summary>Answer</summary>In a shared-nothing architecture, each node is independent — it has its own CPU, memory, and storage, and doesn't share these resources with other nodes. Nodes communicate only via the network. This is the gold standard for horizontal scalability because: (1) no resource contention between nodes, (2) adding nodes adds capacity linearly, (3) failure of one node doesn't affect others, (4) no central bottleneck. Examples: Cassandra, Kafka, application servers behind a load balancer. The opposite is shared-disk or shared-memory architectures (e.g., Oracle RAC), which are harder to scale because of contention on shared resources.</details>

## Scenario-Based Questions
41. Your application works perfectly for 1,000 users but response times spike at 10,000 users. Walk through your debugging approach.

<details><summary>Answer</summary>(1) **Check metrics** — CPU, memory, disk I/O, network on all servers. Identify which component is saturated. (2) **Check the database** — is it the bottleneck? Look at query times, connection counts, slow query logs. (3) **Check for stateful servers** — are sticky sessions causing hotspots? One server might be handling 80% of traffic. (4) **Check caching** — is caching in place? What's the cache hit rate? A low hit rate means most requests hit the database. (5) **Profile hot paths** — use APM tools to find which endpoints are slowest. (6) **Load test incrementally** — find the exact breaking point (e.g., 5,000 users? 7,500?) to understand the degradation curve. Start with the simplest fix: often adding caching or optimizing a slow query solves 80% of the problem.</details>

42. You're a startup with 10,000 users. Your CTO asks you to design the system to handle 1 million users. What do you propose?

<details><summary>Answer</summary>Don't over-engineer upfront. Propose a phased approach: **Phase 1 (10K–100K)**: Separate the database from the application server so each can be tuned independently. Add basic monitoring. **Phase 2 (100K–500K)**: Add a Redis cache for hot data (profiles, sessions, feed). This alone can reduce database load by 80–90%. **Phase 3 (500K–2M)**: Add a load balancer and multiple stateless app servers with auto-scaling. Move sessions to Redis. **Phase 4 (2M+)**: Add read replicas for the database. Consider sharding if writes become a bottleneck. Add a CDN for static assets. Each phase addresses the specific bottleneck that emerges at that scale.</details>

43. Your database read replicas are showing 5 seconds of replication lag. How does this affect the system and what do you do?

<details><summary>Answer</summary>**Impact**: Users may see stale data — e.g., they update their profile but the next page load (from a replica) shows the old profile. Write-then-read scenarios fail silently. This breaks user expectations even though no error occurs. **Fixes**: (1) **Read-your-writes consistency** — after a write, route the user's subsequent reads to the primary for a few seconds. (2) **Reduce replica lag** — check if the replica is under-provisioned (CPU/IO), optimize heavy replication queries, reduce write volume. (3) **Show staleness indicators** — "last updated 5s ago." (4) **Use synchronous replication for critical reads** — at the cost of write latency. (5) **Investigate root cause** — 5s lag is abnormally high; check for long-running queries, schema changes, or network issues blocking replication.</details>

44. Your single Redis cache instance is running out of memory. How do you scale it?

<details><summary>Answer</summary>(1) **Immediate**: Review TTLs — are entries living too long? Reduce TTL for less critical data. Remove unused keys. Use more memory-efficient data structures (hashes instead of individual keys). (2) **Vertical scaling**: Move to a larger Redis instance with more RAM. Simple and fast. (3) **Horizontal scaling**: Deploy Redis Cluster, which automatically partitions data across multiple nodes using hash slots. Data is distributed, and each node handles a subset. (4) **Tiered caching**: Hot data in Redis, warm data in a less expensive cache or evicted to disk. (5) **Eviction policy**: Configure `allkeys-lru` or `volatile-lru` to automatically evict least recently used keys when memory is full.</details>

45. During a flash sale, your system receives 50x normal traffic. What scalability strategies should be in place?

<details><summary>Answer</summary>(1) **Pre-scale** — spin up extra app servers and cache capacity before the event. Don't rely solely on auto-scaling (it takes minutes to react). (2) **CDN** — serve all static assets from CDN to reduce origin load. (3) **Queue writes** — use a message queue for order processing; acknowledge the order immediately, process asynchronously. (4) **Rate limiting** — limit per-user request rates to prevent abuse. (5) **Cache aggressively** — product pages, inventory counts (with short TTL). (6) **Degrade gracefully** — disable non-essential features (recommendations, analytics) to save resources for checkout. (7) **Circuit breakers** — prevent cascading failures if a downstream service buckles. (8) **Load shedding** — reject excess requests with a "please try again" rather than letting the system crash.</details>

46. Your message queue consumers can't keep up with the producers. What do you do?

<details><summary>Answer</summary>(1) **Add more consumers** — this is the primary scaling lever for message queues. Kafka allows adding consumers up to the number of partitions. (2) **Add more partitions** — if consumer count equals partition count, add more partitions to enable more parallelism. (3) **Optimize consumer processing** — reduce per-message processing time (batch database writes, reduce external calls). (4) **Use batch consumption** — consume multiple messages per poll and process them together. (5) **Backpressure** — slow down producers if consumers are overwhelmed. (6) **Dead letter queue** — move poison messages that repeatedly fail to a separate queue instead of blocking the main queue. (7) **Check for consumer bottlenecks** — is the consumer CPU-bound, I/O-bound, or waiting on a downstream service?</details>

47. You need to shard a database that currently has 500 GB of data and 10,000 writes/second. How do you approach this?

<details><summary>Answer</summary>(1) **Choose a shard key** — analyze query patterns. If most queries are per-user, `user_id` is a good choice. Ensure high cardinality and even distribution. (2) **Determine shard count** — consider current size and growth. 4–8 shards might suffice for 500 GB, each shard holding 60–125 GB. Plan for 2–3 years of growth. (3) **Migration strategy** — dual-write: write to both old and new shards during transition. Backfill historical data. Verify consistency. (4) **Handle cross-shard queries** — redesign queries that span shards (denormalize or use a secondary index service). (5) **Routing layer** — build or use a shard router (e.g., Vitess for MySQL). (6) **Test thoroughly** — load test each shard independently and the system end-to-end before cutover.</details>

48. Your application is globally distributed but users in Asia experience 3x higher latency than users in the US. What do you do?

<details><summary>Answer</summary>(1) **Deploy in an Asian region** — spin up app servers and caches in a Singapore/Tokyo data center. (2) **CDN** — ensure your CDN has edge nodes in Asia serving static content. (3) **GeoDNS** — route Asian users to the Asian deployment. (4) **Database read replicas** — place read replicas in the Asian region to avoid cross-Pacific database queries. (5) **Verify routing** — check that DNS and CDN are actually directing Asian users locally (misconfigured routing is common). (6) **Local writes** — for write-heavy operations, consider a multi-region primary or accept asynchronous replication from a US primary. (7) **Reduce payload sizes** — compress responses, optimize images, minimize unnecessary data transfers over the high-latency link.</details>

49. After adding 3 more app servers, throughput didn't increase. Why?

<details><summary>Answer</summary>The bottleneck is not the app tier. Common reasons: (1) **Database is the bottleneck** — all app servers share the same database; adding app servers just increases database load. Check database CPU, connections, and query latency. (2) **Shared cache at capacity** — the Redis cache is maxed out, and more app servers just cause more cache misses or contention. (3) **External dependency** — a third-party API or microservice is the bottleneck. (4) **Network bandwidth** — the network link between app servers and the database is saturated. (5) **Load balancer is the bottleneck** — the LB itself can't handle more traffic. The fix: identify the actual bottleneck using monitoring, then scale that component.</details>

50. Your team is debating whether to use a NoSQL database for a new feature. What factors should drive the decision?

<details><summary>Answer</summary>**Choose NoSQL when**: (1) You need horizontal write scalability beyond what a single SQL primary can handle. (2) Your data model is denormalized or document-oriented (e.g., product catalogs, user profiles). (3) You can tolerate eventual consistency. (4) Your schema is flexible or evolving rapidly. (5) You need very high throughput for simple key-value lookups. **Stick with SQL when**: (1) You need strong consistency and ACID transactions (e.g., financial data). (2) Your queries involve complex joins, aggregations, or ad-hoc analysis. (3) Your team has strong SQL expertise. (4) Data relationships are central to the application (e.g., social graph with complex traversals). Often the answer is "both" — use SQL for transactional data and NoSQL for high-throughput, denormalized workloads.</details>

## Interview-Focused Questions
51. Walk through how you would scale a social media application from 0 to 10 million users.

<details><summary>Answer</summary>**Stage 1 (0–10K users)**: Single server with the app and database on one machine. Simple, cheap, and sufficient. **Stage 2 (10K–100K)**: Separate the database onto its own server. Each component can be tuned independently. **Stage 3 (100K–500K)**: Add Redis caching for hot data (profiles, feeds, sessions). 80–90% of reads are served from cache. **Stage 4 (500K–2M)**: Add a load balancer with multiple stateless app servers. Auto-scale based on CPU/request count. Store sessions in Redis. **Stage 5 (2M–10M)**: Add database read replicas. Primary handles writes, replicas handle reads (10:1+ read-to-write ratio). **Stage 6 (10M+)**: Shard the database by user ID. Each shard handles a subset of users. Add CDN for static content, message queues for async processing (notifications, feed generation). Each stage solves the specific bottleneck that emerges from growth.</details>

52. In a system design interview, how would you explain the trade-offs between vertical and horizontal scaling?

<details><summary>Answer</summary>"Vertical scaling is like getting a bigger truck — it's simple, requires no route changes, but there's a maximum truck size. Horizontal scaling is like adding more trucks — there's no limit on the fleet size, but you need traffic management (load balancer), coordination (data consistency), and each truck must be self-sufficient (stateless). I'd start with vertical scaling for databases and early-stage systems because it's simpler and avoids distributed system complexity. But I'd architect for horizontal scaling from day one — keeping application servers stateless, using external session stores, and designing APIs that don't assume a single server. When vertical limits are reached or fault tolerance is needed, the transition to horizontal scaling is then straightforward."</details>

53. An interviewer asks: "Your database is the bottleneck. How do you scale it?" Walk through your answer.

<details><summary>Answer</summary>"First, I'd identify whether it's a read or write bottleneck. **If read-heavy** (which most apps are): add read replicas. The primary handles writes, replicas handle reads. This is the simplest approach and works for 10:1+ read-to-write ratios. Also add caching — Redis can handle 100K+ ops/sec and serve most hot queries without touching the database. **If write-heavy**: sharding is needed. I'd partition data by a high-cardinality key like user_id, distributing writes across multiple primaries. **If both**: combine sharding with read replicas per shard. **If the data model allows**: consider NoSQL databases like Cassandra or DynamoDB that are designed for horizontal scaling with built-in sharding. I'd also optimize before scaling — add indexes, optimize slow queries, use connection pooling — because often the database isn't truly at capacity, it's being used inefficiently."</details>

54. How would you explain why "adding more servers" doesn't always solve scalability problems?

<details><summary>Answer</summary>"Adding more servers only helps if the application tier is the bottleneck. If the database, cache, or a downstream service is the constraint, more app servers just increase load on the real bottleneck, potentially making things worse. This is Amdahl's Law in practice — the system can only scale as fast as its slowest component. For example, if 100 app servers all query the same database, adding 50 more just means 150 servers hammering the same database. You need to scale the actual bottleneck. Also, adding servers increases distributed system complexity — more network calls, harder debugging, potential consistency issues. The right approach is: (1) identify the bottleneck with monitoring, (2) scale specifically that component, (3) repeat."</details>

55. How would you design a system to handle both predictable daily traffic patterns and unexpected viral spikes?

<details><summary>Answer</summary>"For **predictable daily patterns**: scheduled auto-scaling — pre-scale up before peak hours (e.g., scale up at 8 AM, scale down at midnight based on historical patterns). For **unexpected viral spikes**: (1) reactive auto-scaling with aggressive scaling policies — scale out quickly when CPU or RPS thresholds are breached. (2) CDN for static content to absorb read traffic. (3) Message queues to buffer writes — accept the request immediately, process asynchronously. (4) Rate limiting to prevent any single source from overwhelming the system. (5) Graceful degradation — disable non-essential features (recommendations, analytics) to protect core functionality. (6) Load shedding — if truly overwhelmed, reject excess requests with a 503 rather than letting the entire system crash. (7) Pre-provisioned headroom — always run at 60-70% capacity so there's room for a 2-3x spike before auto-scaling kicks in."</details>

56. An interviewer asks: "What's the difference between scalability and performance?" How do you answer?

<details><summary>Answer</summary>"Performance is how fast a single request is handled — measured in response time (e.g., 50ms). Scalability is how well the system maintains performance as load increases. You can have a high-performance system that doesn't scale (a single-threaded server processing requests in 1ms but limited to 1,000 RPS) and a scalable system with moderate per-request performance (20ms per request but handling 100,000 RPS across many servers). Improving performance (optimizing a query from 100ms to 10ms) often improves scalability too (frees up resources for more requests). But they're different goals requiring different strategies. You optimize performance with caching, query optimization, and algorithm improvements. You improve scalability with horizontal scaling, sharding, and async processing."</details>

57. How would you handle a situation where your database shards become unevenly distributed over time?

<details><summary>Answer</summary>"Uneven shards (data skew) happen because data doesn't grow uniformly — some users generate more data, some shards get more writes. My approach: (1) **Monitor** — track per-shard size, query rate, and CPU/IO utilization. Alert when any shard exceeds 70% of capacity or is 2x the median. (2) **Rebalance** — split the large shard into two by migrating a portion of its key range to a new shard. This requires a migration tool that copies data while the original shard continues serving traffic (dual-read/dual-write pattern). (3) **Consider hash-based sharding** — if using range-based sharding, a hash-based approach distributes data more evenly. (4) **Use a managed solution** — tools like Vitess (MySQL) or CockroachDB handle automatic rebalancing. (5) **Prevention** — choose shard keys with high cardinality and uniform distribution from the start."</details>

58. Your system uses microservices. How do scaling considerations differ from a monolithic architecture?

<details><summary>Answer</summary>"In a monolith, you scale the entire application — even if only the image processing module is the bottleneck. Microservices let you scale individual components independently: scale the image service to 20 instances while keeping the auth service at 3. **Advantages**: targeted scaling based on each service's load, independent deployments, technology diversity (use Go for high-throughput, Python for ML). **Challenges**: (1) network overhead — every service call is a network hop. (2) Data consistency — each service may have its own database, making distributed transactions hard. (3) Cascading failures — one slow service can block others. (4) Operational complexity — monitoring, deploying, and debugging 50 services is harder than one app. (5) Service discovery and load balancing per-service. Microservices are a scaling enabler but also a complexity multiplier."</details>

59. How would you decide between adding read replicas vs caching to handle increased read traffic?

<details><summary>Answer</summary>"Both reduce database read load, but they solve different problems. **Choose caching first when**: (1) the data is frequently accessed and rarely changes (user profiles, product listings) — cache hit rate will be high. (2) You need sub-millisecond response times. (3) The same queries are repeated across many users. Caching gives 100x throughput improvement and massive latency reduction. **Choose read replicas when**: (1) query patterns are diverse — many unique queries that don't benefit from caching. (2) You need full SQL capability on the read path (complex joins, ad-hoc queries). (3) Data freshness is critical — replicas are only slightly behind the primary, while cached data can be staler. **Often, use both**: cache the most popular queries, and use replicas for the long tail of less frequent queries."</details>

60. How do you ensure a distributed system scales without sacrificing data consistency?

<details><summary>Answer</summary>"It depends on how much consistency you truly need — the CAP theorem tells us we can't have perfect consistency, availability, and partition tolerance simultaneously. My approach: (1) **Classify data by consistency requirements** — financial transactions need strong consistency; view counts can tolerate eventual consistency. (2) **For strong consistency**: use a single primary for writes, synchronous replication to replicas (at the cost of write latency), or consensus-based distributed databases (CockroachDB, Spanner) that handle consistency across shards. (3) **For eventual consistency**: use async replication, resolve conflicts with last-write-wins or application-level merge logic. (4) **Hybrid approach**: strong consistency for critical paths (payments, inventory deductions) and eventual consistency for non-critical paths (feed generation, notifications). This is how most large-scale systems actually work."</details>

61. An interviewer presents: "We have 50,000 writes/second and a single primary database. It can't keep up. What do you do?"

<details><summary>Answer</summary>"Step-by-step: (1) **Optimize first** — batch inserts (100 rows per INSERT instead of 100 individual INSERTs), reduce indexes on write-heavy tables, use async writes where possible. This alone can 2-5x write throughput. (2) **Buffer writes with a message queue** — Kafka can absorb millions of writes/second. Consumers batch-write to the database at a sustainable rate. This smooths out spikes and decouples write acceptance from write processing. (3) **Shard the database** — if sustained 50K writes/sec exceeds one primary's capacity, shard by a key that distributes writes evenly. 5 shards means each handles 10K writes/second. (4) **Consider a write-optimized database** — Cassandra, ScyllaDB, or DynamoDB are designed for high write throughput. (5) **CQRS (Command Query Responsibility Segregation)** — separate the write model from the read model, optimizing each independently."</details>

62. How would you scale a real-time chat application to millions of concurrent users?

<details><summary>Answer</summary>"(1) **Connection layer**: Use WebSocket servers — each server holds thousands of persistent connections. Scale horizontally by adding more WebSocket servers behind a load balancer with sticky connections (by connection ID, not user session). (2) **Message routing**: Use a pub/sub system (Redis Pub/Sub, Kafka) to route messages between servers. When User A on Server 1 messages User B on Server 3, Server 1 publishes to a channel, and Server 3 receives and delivers. (3) **Presence tracking**: Store online status in Redis with TTLs. (4) **Message storage**: Write messages to a sharded database (shard by conversation ID). Use Cassandra or DynamoDB for high write throughput. (5) **Group chats**: Fan-out on read (each user fetches their unread messages) or fan-out on write (write a copy per recipient). Fan-out on read scales better for large groups. (6) **Geographic distribution**: Deploy in multiple regions with local pub/sub and cross-region sync."</details>

63. What questions should you ask an interviewer before proposing a scaling solution?

<details><summary>Answer</summary>"(1) **What's the current scale?** — users, RPS, data volume, growth rate. Are we scaling from 1K to 100K or from 1M to 100M? (2) **What's the bottleneck?** — is it reads, writes, storage, compute, or network? (3) **What are the latency requirements?** — sub-100ms for real-time apps vs seconds for batch processing? (4) **What's the read-to-write ratio?** — 100:1 means focus on read scaling; 1:1 means write scaling matters. (5) **What consistency guarantees are needed?** — strong consistency (banking) vs eventual (social media likes)? (6) **What's the budget?** — can we use managed cloud services? (7) **What's the team's expertise?** — sharding a database requires specific skills. (8) **What's the data model?** — relational or document? Are there joins? These questions show the interviewer you think systematically rather than jumping to a solution."</details>

64. How would you explain database sharding to a non-technical stakeholder?

<details><summary>Answer</summary>"Imagine a library with one librarian. When the library has 100 books, one librarian handles everything fine. But with 1 million books, one librarian can't serve all visitors fast enough, and there's not enough shelf space. Sharding is like splitting the library into 10 branch libraries — each holds a portion of the books (e.g., by author's last name: A-C in Branch 1, D-F in Branch 2). Each branch has its own librarian and shelves. Now the system handles 10x more visitors in parallel. The trade-off: if someone wants to compare books across all branches, a librarian from each branch must coordinate — that's slower. So we design the split carefully to minimize these cross-branch requests."</details>

65. Describe a scenario where you would scale vertically first and then transition to horizontal scaling.

<details><summary>Answer</summary>"Consider a startup's PostgreSQL database handling 5,000 QPS. The server is a 4-core, 16 GB machine. **Vertical scaling first**: upgrade to a 32-core, 256 GB machine. This 10x upgrade is done in a few hours with minimal risk, no code changes, and the database can now cache most of the dataset in memory. This buys 1–2 years of growth time. **Transition to horizontal**: at 50,000 QPS and 500 GB of data, even the biggest available machine struggles. Now add read replicas for the read-heavy workload (20:1 read-to-write ratio). This handles up to 200,000 QPS reads. When writes become the bottleneck at 20,000 writes/second, shard the database. The key insight: vertical scaling bought time to build the expertise and infrastructure needed for horizontal scaling, rather than prematurely introducing distributed system complexity."</details>

66. What are the scalability implications of choosing synchronous vs asynchronous communication between services?

<details><summary>Answer</summary>"**Synchronous (HTTP/gRPC)**: The caller waits for the response. This creates tight coupling — if the downstream service is slow, the caller's thread is blocked, and threads pile up. Under high load, this causes cascading failures and limits scalability. Throughput is bounded by the slowest service in the chain. **Asynchronous (message queues)**: The caller publishes a message and moves on. The consumer processes it later. This decouples services — the producer and consumer scale independently. Traffic spikes are absorbed by the queue. But it introduces eventual consistency (the work isn't done when the caller returns) and adds complexity (monitoring queue depth, handling failures). **Best practice**: use synchronous for queries where the user needs an immediate answer (read APIs) and asynchronous for commands/mutations that can be processed later (sending emails, generating reports, updating analytics)."</details>

67. How does data denormalization help with read scalability and what are the trade-offs?

<details><summary>Answer</summary>"Normalization avoids data duplication by storing each fact once, but it requires joins to reconstruct data — joins don't scale well across shards. Denormalization pre-joins data by storing redundant copies optimized for specific read patterns. For example, instead of joining `users`, `Orders`, and `products` tables, store a denormalized `OrderSummary` with user name, product name, and price in one document. **Pros**: fast single-table reads, no joins, sharding-friendly, cache-friendly. **Cons**: data redundancy (storage cost), update complexity (changing a user's name requires updating all their order summaries), risk of inconsistency if updates are missed. Denormalization is a trade-off: optimize reads at the expense of write complexity and storage. It's essential at scale and is the default data model in NoSQL databases."</details>

68. In a system design interview, how would you discuss the scalability of a payment processing system?

<details><summary>Answer</summary>"Payment systems have unique scalability constraints: (1) **Strong consistency is mandatory** — you can't have eventual consistency for money. No double-charges or lost payments. (2) **Database**: start with a single primary with synchronous replication for durability. Shard by merchant_id or region when single-primary write throughput is exceeded. (3) **Idempotency**: every payment operation must be idempotent (using idempotency keys) so retries don't cause duplicate charges. (4) **Queue writes**: accept payment requests into a durable queue (Kafka with replication), process them serially per-user to avoid race conditions. (5) **Rate limiting**: protect against fraud and abuse. (6) **Geographic concerns**: process payments close to the payment gateway for lower latency and regulatory compliance. (7) **Scaling**: payments are write-heavy — cache doesn't help much. Focus on write-path optimization: batching, connection pooling, and sharding."</details>

69. How do you scale a search feature across billions of documents?

<details><summary>Answer</summary>"(1) **Inverted index**: Build an inverted index mapping every word to the documents containing it (this is how Elasticsearch/Lucene works). (2) **Sharding**: Partition the index across multiple nodes — either by document (each shard holds a subset of documents) or by term (each shard holds a subset of the vocabulary). Document-based sharding is more common. (3) **Replication**: Replicate each shard for read scalability and fault tolerance. (4) **Scatter-gather**: A search query goes to every shard, each returns its top results, and a coordinator merges and ranks them. (5) **Caching**: Cache popular queries (search results for "iPhone" don't change every second). (6) **Tiered indexing**: Hot data (recent documents) in memory, cold data on SSD. (7) **Elasticsearch/Solr clusters**: These tools handle sharding, replication, and querying out of the box. Scale by adding more data nodes."</details>

70. What scalability patterns would you apply when designing a system that handles both real-time and batch workloads?

<details><summary>Answer</summary>"This is the Lambda or Kappa architecture problem. **Lambda architecture**: Two separate paths — a real-time path (stream processing with Kafka + Spark Streaming or Flink) for low-latency queries on recent data, and a batch path (Spark/Hadoop) for comprehensive processing of all historical data. Results are merged in a serving layer. **Kappa architecture**: simplifies by using a single stream processing system for both real-time and batch (replay the stream for batch). **Scaling each**: (1) Stream path: scale Kafka partitions and stream processor instances. (2) Batch path: scale compute nodes. (3) Serving layer: use a sharded, replicated database (Cassandra, Elasticsearch). **Key insight**: separate the ingestion, processing, and serving layers so each scales independently. The real-time path needs low latency; the batch path needs high throughput — different optimization goals."</details>

71. How do you decide when it's time to move from a monolith to microservices for scalability?

<details><summary>Answer</summary>"Don't switch for theoretical benefits. Switch when you have concrete pain: (1) **Scaling friction**: you need to scale the image processing module 10x but can't without scaling the entire monolith. (2) **Deployment bottleneck**: deploying the monolith takes hours and a small change risks the entire system. Teams are blocked waiting for each other. (3) **Team scaling**: when the team grows beyond ~10–15 engineers, a monolith creates merge conflicts and ownership confusion. (4) **Technology constraints**: a module needs a language or framework that the monolith doesn't support. **Warning signs it's premature**: (1) you have fewer than 5 engineers, (2) you don't have operational maturity (monitoring, CI/CD, containerization), (3) service boundaries aren't clear yet. Extract services incrementally — start with the most obvious bottleneck component, not a big-bang rewrite."</details>

72. An interviewer asks: "Your system needs to support 100,000 concurrent WebSocket connections. How do you scale this?"

<details><summary>Answer</summary>"(1) **Per-server capacity**: a single server can handle 50K–100K+ WebSocket connections with proper tuning (increase file descriptor limits, use epoll/kqueue, optimize memory per connection — each connection uses ~10-50 KB). (2) **Horizontal scaling**: 10 servers can handle 500K–1M connections. Use a load balancer with connection-aware routing (not round-robin per request — WebSocket is long-lived). (3) **Cross-server messaging**: when user A on Server 1 needs to message user B on Server 3, use a pub/sub system (Redis Pub/Sub, Kafka). Each server subscribes to channels for its connected users. (4) **State management**: track which user is on which server in Redis. On reconnect, route to any available server and re-subscribe. (5) **Graceful degradation**: if a server goes down, clients reconnect to other servers and re-subscribe. (6) **Health monitoring**: track connections per server, message latency, and reconnection rates."</details>

73. How would you scale a notification system that sends 10 million push notifications per hour?

<details><summary>Answer</summary>"10M per hour = ~2,800 per second. (1) **Ingestion**: accept notification requests via Kafka with multiple partitions. Producers are the services triggering notifications. (2) **Processing**: multiple consumer instances read from Kafka partitions. Each consumer batches notifications by device type (iOS, Android, Web). (3) **Delivery**: use platform-specific APIs — APNs for iOS, FCM for Android. Both support batched delivery (up to 500–1000 per request). (4) **Rate limiting per platform**: APNs and FCM have rate limits; implement per-connection throttling. (5) **Retry with backoff**: failed deliveries go to a retry queue with exponential backoff. (6) **Device token management**: maintain a database of device tokens, sharded by user_id. Handle token invalidation from APNs/FCM feedback. (7) **Scale**: 2,800/sec is manageable with 5–10 consumer instances, each making batched API calls. For 100M+/hour, add more partitions and consumers linearly."</details>

74. What is the difference between scaling for throughput vs scaling for storage?

<details><summary>Answer</summary>"**Scaling for throughput** means handling more operations per second — more reads, writes, API calls. Solutions: add more app servers, add read replicas, use caching, partition data across shards to parallelize work. The goal is to process more requests without increasing response time. **Scaling for storage** means accommodating more data — larger datasets, more users, more history. Solutions: add more disk/storage, shard the database to distribute data across machines, archive old data, compress data, use tiered storage (hot data on SSD, cold on HDD/S3). Often both grow together, but they require different strategies. You might need to scale storage (1 TB → 10 TB) without needing more throughput, or scale throughput (1K → 100K RPS) without more storage."</details>

75. How does data partitioning strategy affect the scalability of analytical queries?

<details><summary>Answer</summary>"If data is partitioned by `user_id` (common for transactional workloads), analytical queries like 'total revenue by product category' must scan every partition — this is a scatter-gather that scales poorly as partitions increase. Solutions: (1) **Separate OLAP store**: replicate data to an analytical database (BigQuery, Redshift, ClickHouse) optimized for column-oriented scans and aggregations. (2) **Different partitioning for analytics**: partition the analytical copy by date or product category to align with common query patterns. (3) **Materialized views**: pre-compute common aggregations and store them in a denormalized table. (4) **Lambda architecture**: maintain separate systems for real-time analytics (stream processing) and historical analytics (batch processing). The key insight: transactional and analytical workloads have conflicting partitioning needs, so separate them."</details>

76. Explain how you would perform a zero-downtime database migration during a scaling operation.

<details><summary>Answer</summary>"(1) **Create the new schema/shards** alongside the existing database. (2) **Dual-write**: modify the application to write to both old and new databases. Verify writes succeed on both. (3) **Backfill**: copy historical data from the old database to the new one using a background migration job. Use timestamps or sequence IDs to track progress. (4) **Verify consistency**: compare data between old and new databases to ensure they match. (5) **Switch reads**: gradually shift read traffic to the new database (canary → percentage-based rollout → 100%). (6) **Monitor**: track error rates, latency, and data consistency throughout. (7) **Remove dual-write**: once confident, stop writing to the old database. (8) **Decommission**: keep the old database as a backup for a grace period, then shut it down. This process can take days to weeks for large datasets."</details>

77. What role does observability play in scaling a system?

<details><summary>Answer</summary>"You cannot scale what you cannot measure. Observability provides the data needed to: (1) **Identify bottlenecks** — metrics show which component is saturated (CPU, memory, DB connections). (2) **Validate scaling decisions** — after adding read replicas, did database CPU drop? Did latency improve? (3) **Trigger auto-scaling** — auto-scaling policies depend on metrics (CPU > 70%, queue depth > 1000). (4) **Detect scaling limits** — anomalous latency spikes or error rate increases reveal when a component can't handle more load. (5) **Capacity planning** — historical trends predict when you'll need to scale. Key tools: metrics (Prometheus/Grafana), distributed tracing (Jaeger), logs (ELK stack), alerting (PagerDuty). Without observability, scaling decisions are guesswork."</details>

78. How do you handle database schema changes at scale without downtime?

<details><summary>Answer</summary>"Schema changes (ALTER TABLE) on large tables can lock the table for hours, causing downtime. Strategies: (1) **Online schema change tools** — pt-online-schema-change (MySQL) or pg_repack (PostgreSQL) create a new table with the new schema, copy data over, then atomically swap. Writes to the original table are captured and replayed. (2) **Expand-contract pattern**: first add the new column (non-breaking), deploy code that writes to both old and new columns, backfill the new column, deploy code that reads from the new column, then remove the old column. (3) **Use schema-less databases** — document databases like MongoDB allow adding fields without ALTER TABLE. (4) **Blue-green deployment**: run two database versions in parallel and switch traffic. Each step must be backward compatible."</details>

79. What are the trade-offs between pre-scaling and reactive scaling?

<details><summary>Answer</summary>"**Pre-scaling (proactive)**: provision extra capacity before you need it. Pros: handles traffic spikes without delay, no risk of being caught off-guard, no auto-scaling lag. Cons: wastes money on idle resources, requires accurate traffic prediction, over-provisioning may be significant. **Reactive scaling (auto-scaling)**: add capacity in response to real-time demand metrics. Pros: cost-efficient (pay for what you use), handles unpredictable patterns, no human intervention needed. Cons: takes 2–5 minutes to spin up new instances (cold start), may not react fast enough for sudden spikes, can oscillate if thresholds are poorly configured. **Best practice**: combine both — pre-scale for known events (Black Friday, product launches), use auto-scaling for daily patterns and unexpected spikes. Always maintain a buffer (never run above 70% utilization)."</details>

80. How would you architect a system that needs to scale from 1,000 to 1,000,000 users without a complete redesign?

<details><summary>Answer</summary>"Design with scalability principles from day one, even if you start small: (1) **Stateless app servers** — store sessions in Redis from the start. (2) **Database abstraction** — use a repository/DAO pattern so switching from a single DB to sharded DBs only changes the data layer. (3) **Externalize configuration** — feature flags, connection strings, and scaling parameters live in config, not code. (4) **API-first design** — well-defined APIs between components mean you can replace any component without changing others. (5) **Async by default** — use message queues for anything that doesn't need an immediate response. (6) **Monitoring from day one** — you can't scale what you can't measure. (7) **Progressive scaling**: single server → separate DB → add cache → add LB + multiple servers → read replicas → sharding. Each step is additive, not a rewrite. The code doesn't change; the infrastructure evolves."</details>
