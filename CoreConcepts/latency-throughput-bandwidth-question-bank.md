# Latency vs Throughput vs Bandwidth — Question Bank

## Beginner Level
1. What is latency in the context of system design?

<details><summary>Answer</summary>Latency is the time it takes for a single request to travel from the source (client) to the destination (server) and back. It measures **delay**, not volume. In networking, this is often called round-trip time (RTT). For example, if a client sends an HTTP request and receives the response 150ms later, the latency is 150ms. Latency includes propagation delay, transmission delay, processing delay, and queuing delay.</details>

2. What is throughput in the context of system design?

<details><summary>Answer</summary>Throughput is the amount of work completed per unit of time. It measures **volume**, not speed. For web systems, throughput is typically expressed as requests per second (RPS) or transactions per second (TPS). For example, a server handling 5,000 HTTP requests per second has a throughput of 5,000 RPS. Throughput tells you how much total work your system can handle, regardless of how fast each individual request completes.</details>

3. What is bandwidth in networking?

<details><summary>Answer</summary>Bandwidth is the **maximum rate** at which data can be transferred over a network link. It measures capacity, not actual usage. Bandwidth is typically expressed in bits per second (bps) — Kbps, Mbps, Gbps. For example, a 1 Gbps Ethernet link has a bandwidth of 1 billion bits per second. Bandwidth is a theoretical ceiling; the actual data transfer rate (throughput) is almost always lower due to protocol overhead, congestion, and other factors.</details>

4. How do latency, throughput, and bandwidth differ in one sentence each?

<details><summary>Answer</summary>**Latency** = How long one request takes (measures delay). **Throughput** = How many requests complete per second (measures volume). **Bandwidth** = The maximum data transfer rate the link supports (measures capacity). An analogy: on a highway, latency is how long one car takes to travel from City A to City B, throughput is how many cars arrive per hour, and bandwidth is the number of lanes available.</details>

5. Explain the highway analogy for latency, throughput, and bandwidth.

<details><summary>Answer</summary>Think of a highway connecting two cities. **Bandwidth** is the number of lanes — more lanes mean more cars can travel simultaneously. **Throughput** is how many cars actually pass through per hour — this depends on traffic conditions, not just the number of lanes. **Latency** is the time it takes for a single car to travel from one city to the other. A highway might have 4 lanes (high bandwidth), but if there's an accident, only 100 cars pass per hour (low throughput). Meanwhile, each car might take 2 hours (high latency). These three metrics don't always move together.</details>

6. What is round-trip time (RTT)?

<details><summary>Answer</summary>RTT (Round-Trip Time) is the total time from when a client sends a request to when it receives the response back. It includes: (1) time for the request to travel from client to server (propagation + transmission), (2) server processing time, and (3) time for the response to travel back. RTT is the most common way to measure network latency. For example, a `ping` command measures RTT — if you ping a server and get 40ms, that's the RTT. Actual application-level latency may be higher than raw RTT because of additional processing, TLS handshakes, and queuing.</details>

7. Why is throughput always less than or equal to bandwidth?

<details><summary>Answer</summary>Bandwidth is the **theoretical maximum** capacity of a link, while throughput is the **actually achieved** data rate. Throughput is always lower because of: (1) protocol overhead — headers, acknowledgments, and control packets consume bandwidth, (2) congestion and packet loss — retransmissions waste capacity, (3) processing limitations — endpoints may not generate or consume data fast enough, (4) flow control and windowing — TCP limits in-flight data, (5) shared links — other traffic competes for the same bandwidth. It's like a highway: 4 lanes (bandwidth) doesn't mean 4 lanes worth of cars arrive per hour (throughput) — traffic lights, accidents, and merging slow things down.</details>

8. What are the four components of network latency?

<details><summary>Answer</summary>(1) **Propagation delay** — time for signals to physically travel through the medium. Light in fiber travels at ~200,000 km/s, so a cross-Atlantic cable (6,000 km) adds ~30ms. (2) **Transmission delay** — time to push all bits of a packet onto the wire. Depends on packet size and link bandwidth. (3) **Processing delay** — time for routers, load balancers, and servers to process packets (inspect headers, make routing decisions, run application logic). (4) **Queuing delay** — time spent waiting in buffers when components are busy. Under high load, queuing delay dominates and causes latency spikes.</details>

9. What units are latency, throughput, and bandwidth typically measured in?

<details><summary>Answer</summary>**Latency** is measured in time units — milliseconds (ms), microseconds (µs), or seconds (s). **Throughput** is measured in work per time — requests per second (RPS), transactions per second (TPS), queries per second (QPS), or megabytes per second (MB/s). **Bandwidth** is measured in bits per second — Kbps (kilobits), Mbps (megabits), Gbps (gigabits). Note the bit vs byte distinction: bandwidth uses bits (b), while throughput often uses bytes (B). 1 byte = 8 bits, so a 1 Gbps link transfers at most 125 MB/s.</details>

10. Can you have high bandwidth but low throughput? Give an example.

<details><summary>Answer</summary>Yes. A 10 Gbps fiber link between two data centers (high bandwidth) might achieve only 100 Mbps actual throughput if the application uses a single TCP connection with a small window size, or if packet loss forces frequent retransmissions. Another example: a 1 Gbps link carrying traffic to a server whose CPU is maxed out — the server can't process incoming packets fast enough, so actual throughput is far below the link's capacity. The bandwidth is there, but bottlenecks elsewhere prevent it from being utilized.</details>

11. Can you have low latency but low throughput? Give an example.

<details><summary>Answer</summary>Yes. A single-threaded web server can process each request in 1ms (low latency) but can only handle one request at a time — giving a throughput of just 1,000 RPS. Even though each individual request is fast, the server has no concurrency, so total volume is limited. Another example: a low-latency in-memory cache responds in microseconds, but if it only has one network interface running at 100 Mbps, its throughput is capped by the link speed despite the fast per-request response time.</details>

12. What is the difference between bandwidth and throughput using a pipe analogy?

<details><summary>Answer</summary>Think of a water pipe. **Bandwidth** is the diameter of the pipe — it determines the maximum volume of water that can flow through at any moment. **Throughput** is the actual volume of water flowing through the pipe per second. The pipe might be wide (high bandwidth), but if the water pressure is low, or if there's a clog somewhere, the actual flow (throughput) will be much less than what the pipe could theoretically carry. You can never push more water than the pipe diameter allows (throughput ≤ bandwidth), but you'll almost always push less.</details>

## Intermediate Level
13. Why do we measure latency using percentiles instead of averages?

<details><summary>Answer</summary>Averages hide outliers. A system with 10ms average latency might have p99 of 500ms — meaning 1% of users experience 50x worse performance. Percentiles give a more honest picture of user experience. Common percentiles: **p50 (median)** — half of requests are faster, **p95** — 95% of requests are faster, **p99** — 99% are faster, **p99.9** — 99.9% are faster. Services with millions of users especially care about p99/p99.9 because even 1% of users experiencing bad performance can mean thousands of unhappy customers. Amazon found that every 100ms increase in p99 latency costs 1% of sales.</details>

14. What is the relationship between latency and throughput in a single-threaded system?

<details><summary>Answer</summary>In a single-threaded system: `Throughput = 1 / Latency`. If each request takes 10ms, the system can complete 1 / 0.01 = 100 requests/second. If latency increases to 50ms, throughput drops to 20 RPS. They are inversely related — reducing latency directly increases throughput. This is because the system can only work on one request at a time, so it must finish one before starting the next. Improving latency (e.g., faster database queries) is the primary way to improve throughput in single-threaded systems.</details>

15. How does adding concurrency (multiple workers) affect the relationship between latency and throughput?

<details><summary>Answer</summary>With concurrency: `Throughput = Concurrent Workers / Latency`. If latency is 10ms and you have 50 workers, throughput = 50 / 0.01 = 5,000 RPS. Adding workers multiplies throughput without changing latency. However, there are diminishing returns — beyond a point, adding workers increases contention (lock contention, context switching, shared resource saturation), which *increases* latency and reduces gains. The optimal number of workers depends on the workload type (CPU-bound vs I/O-bound) and available resources. I/O-bound workloads benefit more from concurrency.</details>

16. What is Little's Law and how does it connect latency, throughput, and concurrency?

<details><summary>Answer</summary>Little's Law: `L = λ × W`, where **L** = average number of requests in the system (concurrency), **λ** = arrival rate / throughput (requests per second), **W** = average latency (time each request spends in the system). For example, if throughput is 1,000 RPS and average latency is 200ms, then L = 1,000 × 0.2 = 200 concurrent requests in the system. This law is universal and tells you: to sustain a given throughput at a given latency, you need enough concurrency (threads, connections, workers). If you want 5,000 RPS at 50ms latency, you need 250 concurrent connections.</details>

17. What is the Bandwidth-Delay Product (BDP) and why does it matter?

<details><summary>Answer</summary>BDP = Bandwidth × Latency (specifically, round-trip time). It represents the maximum amount of data that can be "in flight" (sent but not yet acknowledged) at any moment. Example: 1 Gbps link with 100ms RTT → BDP = 125 MB/s × 0.1s = 12.5 MB. This means 12.5 MB of data can exist in the "pipe" simultaneously. BDP matters because TCP's sliding window must be at least as large as the BDP to fully utilize the link. If your TCP window size is only 64 KB but BDP is 12.5 MB, you're using less than 1% of the available bandwidth, resulting in terrible throughput despite high bandwidth.</details>

18. How does TCP window size relate to bandwidth utilization?

<details><summary>Answer</summary>TCP uses a sliding window to control how much data can be in flight (sent but unacknowledged). If the window is smaller than the Bandwidth-Delay Product (BDP), the sender must pause and wait for ACKs before sending more data, leaving the link idle. For example, with a 1 Gbps link and 100ms RTT, BDP = 12.5 MB. If the TCP window is only 64 KB, the sender sends 64 KB, then waits ~100ms for the ACK. Effective throughput = 64 KB / 0.1s = 640 KB/s — only 0.05% of the 125 MB/s capacity. TCP window scaling (RFC 1323) allows windows up to 1 GB, which is essential for high-bandwidth, high-latency links (long fat networks).</details>

19. What is a "long fat network" (LFN) and why is it challenging?

<details><summary>Answer</summary>A Long Fat Network (LFN, pronounced "elephant") is a network path with high bandwidth and high latency — resulting in a large Bandwidth-Delay Product (BDP). Example: a 10 Gbps transcontinental link with 80ms RTT has BDP = 1.25 GB/s × 0.08s = 100 MB. LFNs are challenging because: (1) TCP needs very large window sizes to keep the pipe full. (2) Packet loss is devastating — losing one packet can cause the sender to halve its window, drastically reducing throughput. (3) Congestion control algorithms (like TCP Reno) recover too slowly on LFNs. Solutions include TCP window scaling, selective acknowledgments (SACK), and modern congestion control algorithms like BBR.</details>

20. What factors affect network latency?

<details><summary>Answer</summary>(1) **Geographic distance** — more distance = more propagation delay (NYC to London ≈ 70ms RTT in fiber). (2) **Network congestion** — overloaded routers cause queuing delays. (3) **Number of hops** — each router/switch adds processing delay. (4) **Server load** — busy servers queue requests, increasing processing time. (5) **DNS resolution** — cold requests need DNS lookup (adds 10–100ms). (6) **TLS handshake** — establishing HTTPS adds 1–2 round trips (50–200ms). (7) **Database queries** — slow queries add latency to the response. (8) **Protocol overhead** — TCP three-way handshake adds one RTT before data transfer begins.</details>

21. What strategies can reduce latency?

<details><summary>Answer</summary>(1) **CDNs** — serve content from edge locations closer to users, reducing propagation delay. (2) **Caching** — cache at multiple layers (browser, CDN, application, database) to eliminate round trips. (3) **Connection pooling** — reuse existing TCP connections, avoiding repeated handshakes. (4) **Database optimization** — add indexes, optimize queries, use read replicas. (5) **Geographic distribution** — deploy servers in multiple regions closer to users. (6) **Protocol optimization** — use HTTP/2 (multiplexing), HTTP/3 (QUIC, 0-RTT), to reduce handshake overhead. (7) **Prefetching/preloading** — anticipate user actions and fetch data before it's requested. (8) **Async processing** — offload non-critical work to background queues.</details>

22. What limits throughput in a system?

<details><summary>Answer</summary>Throughput is limited by the **bottleneck** — the system's slowest or most constrained component. Common limiters: (1) **CPU** — compute-bound tasks (encryption, compression, JSON serialization). (2) **Memory** — large working sets cause cache misses or swapping. (3) **I/O** — slow disk reads/writes or network saturation. (4) **Database** — slow queries, lock contention, connection pool exhaustion. (5) **Connection limits** — max open file descriptors, max DB connections. (6) **Contention** — locks, mutexes, shared resources create serialization points. (7) **External dependencies** — slow third-party APIs block progress. The system is only as fast as its slowest component.</details>

23. How can you improve throughput?

<details><summary>Answer</summary>(1) **Horizontal scaling** — add more servers behind a load balancer. (2) **Vertical scaling** — add more CPU, memory, faster SSDs. (3) **Async processing** — use message queues (Kafka, RabbitMQ) to decouple slow operations. (4) **Batching** — process multiple items together (batch inserts, bulk API calls). (5) **Caching** — reduce repeated work by caching results (Redis, Memcached). (6) **Connection pooling** — reuse expensive DB and HTTP connections. (7) **Load balancing** — distribute work evenly across servers. (8) **Reducing latency** — lower per-request latency directly increases single-worker throughput. (9) **Partitioning/sharding** — split data across nodes to parallelize work.</details>

24. What are the different types of bandwidth in a system?

<details><summary>Answer</summary>(1) **Network bandwidth** — capacity of network links (e.g., 1 Gbps Ethernet, 10 Gbps fiber). (2) **Memory bandwidth** — rate of data transfer to/from RAM (e.g., DDR4: ~25 GB/s, DDR5: ~50 GB/s). (3) **Disk bandwidth** — read/write speed of storage (HDD: ~100–200 MB/s, SSD: ~500 MB/s, NVMe: ~3–7 GB/s). (4) **Bus bandwidth** — internal data transfer rate (PCIe 4.0 x16: ~64 GB/s). Each type can become a bottleneck. A system with 10 Gbps network but HDD storage bottlenecks on disk I/O. Understanding which type of bandwidth is the constraint is key to optimization.</details>

25. How does a CDN reduce latency?

<details><summary>Answer</summary>A CDN (Content Delivery Network) places copies of content on **edge servers** distributed globally, closer to end users. Instead of every request traveling to the origin server (e.g., US East Coast), users connect to the nearest edge server (e.g., in Mumbai, London, Tokyo). This reduces: (1) **Propagation delay** — shorter physical distance. (2) **Number of hops** — fewer intermediate routers. (3) **TLS handshake time** — handshake with the nearby edge server is faster. (4) **Origin server load** — fewer requests reach the origin, reducing its latency and increasing its throughput. A user in India fetching from a Mumbai edge (5ms) vs a US origin (200ms) sees a 40x improvement in network latency.</details>

26. How does HTTP/2 improve latency and throughput compared to HTTP/1.1?

<details><summary>Answer</summary>HTTP/1.1 problems: (1) head-of-line blocking — one slow response blocks all subsequent requests on the same connection. (2) Multiple TCP connections required for parallelism (typically 6 per domain). HTTP/2 solutions: (1) **Multiplexing** — multiple requests/responses interleaved on a single TCP connection, eliminating head-of-line blocking at the HTTP level. (2) **Header compression (HPACK)** — reduces overhead from repetitive headers. (3) **Server push** — server proactively sends resources before the client requests them. (4) **Stream prioritization** — important resources load first. Result: fewer TCP connections needed (saves handshake latency), better utilization of existing connections (higher throughput), and reduced overall page load time.</details>

27. How does HTTP/3 (QUIC) further improve latency?

<details><summary>Answer</summary>HTTP/3 replaces TCP with **QUIC** (a UDP-based protocol). Key improvements: (1) **0-RTT connection establishment** — returning clients can send data immediately without any handshake (TCP + TLS requires 2–3 RTTs). (2) **No head-of-line blocking at the transport layer** — in HTTP/2 over TCP, a single lost packet blocks ALL streams. QUIC handles streams independently, so a lost packet on stream A doesn't block stream B. (3) **Connection migration** — QUIC connections survive IP changes (e.g., switching from WiFi to cellular). (4) **Built-in encryption** — TLS 1.3 is integrated into the protocol, so there's no separate TLS handshake. This is especially impactful on high-latency, lossy mobile networks.</details>

28. What is queuing delay and when does it dominate latency?

<details><summary>Answer</summary>Queuing delay is the time a request spends waiting in buffers before being processed. It occurs at routers, load balancers, servers, and database connection pools. Under low load, queuing delay is near zero. Under high load (approaching capacity), it grows **exponentially** — this is described by queuing theory (M/M/1 queue model). When utilization exceeds ~70–80%, queuing delay spikes dramatically. For example, at 90% utilization, average queue length grows 9x; at 99%, it grows 99x. This is why systems should not run near 100% capacity — the tail latency (p99) becomes unbearable even though average latency seems manageable.</details>

29. How does DNS resolution contribute to latency?

<details><summary>Answer</summary>When a client first connects to a domain, it must resolve the domain name to an IP address via DNS. This involves: (1) checking the local DNS cache (0ms if cached), (2) querying the recursive resolver (ISP's DNS or public DNS like 8.8.8.8), (3) potentially querying root → TLD → authoritative name servers (adds 50–200ms if not cached anywhere). DNS adds latency only on the **first request** to a domain (cold start); subsequent requests use the cached result (governed by TTL). Mitigations: low DNS TTL for quick failover, DNS prefetching (`<link rel="dns-prefetch">`), and using persistent connections to avoid re-resolving.</details>

30. What is the impact of TLS handshakes on latency?

<details><summary>Answer</summary>TLS 1.2 requires **2 additional round trips** before data can be exchanged: (1) TCP three-way handshake (1 RTT), (2) TLS handshake — client hello, server hello + certificate, key exchange, finished (1–2 RTTs). On a 100ms RTT link, this adds 200–300ms before the first byte of application data. TLS 1.3 reduces this to **1 RTT** (combining key exchange steps). With **0-RTT resumption** (TLS 1.3), returning clients can send data on the very first packet. HTTP/3 (QUIC) integrates TLS 1.3, achieving 0-RTT for returning clients and 1-RTT for new ones. For APIs with short-lived connections, TLS overhead can dominate total latency.</details>

## Advanced Level
31. Explain how latency and throughput can be optimized independently, with examples.

<details><summary>Answer</summary>**Optimizing latency without affecting throughput**: Adding a CDN reduces latency for individual users but doesn't change the origin server's throughput capacity. Using precomputed caches (e.g., preloading product pages into Redis) makes individual reads faster without changing how many requests the system can handle overall. **Optimizing throughput without affecting latency**: Horizontal scaling (adding more servers behind a load balancer) increases total system throughput (more requests handled in parallel) but doesn't make any individual request faster. Batching database inserts (100 rows in one query vs 100 separate queries) increases throughput but each batch operation may have higher individual latency. These are independent optimization axes.</details>

32. A system has a p50 latency of 5ms and p99 latency of 2 seconds. What does this tell you?

<details><summary>Answer</summary>This tells you the system has a **massive tail latency problem**. Most users (50%) experience excellent performance (< 5ms), but 1% of requests take 400x longer (2 seconds). This suggests: (1) intermittent resource contention — GC pauses, lock contention, or thread pool exhaustion hitting a fraction of requests. (2) Some requests trigger expensive code paths — e.g., cache misses that fall through to a slow database query. (3) External dependency timeouts — 1% of requests may hit a slow third-party service. (4) Noisy neighbors — in shared infrastructure, occasional resource contention spikes. You should investigate: GC logs, slow query logs, thread dumps during high-p99 periods, and dependency call latency breakdowns.</details>

33. How does connection pooling reduce latency?

<details><summary>Answer</summary>Without connection pooling, every request requires creating a new TCP connection (1 RTT for handshake) and possibly a TLS handshake (1–2 more RTTs), plus authentication (e.g., database login). For a database call over a 5ms network: TCP handshake (5ms) + TLS (10ms) + auth (5ms) = 20ms overhead before the query even starts. With connection pooling, connections are established once and reused across requests. The overhead drops to nearly 0ms for subsequent requests. For applications making hundreds of DB calls per second, this saves enormous cumulative latency. Connection pools also prevent connection exhaustion — reusing 50 connections instead of creating 5,000/second.</details>

34. What is tail latency amplification in microservices architectures?

<details><summary>Answer</summary>In microservices, a single user request may fan out to multiple downstream services. The overall latency is determined by the **slowest** downstream call. Even if each service has a p99 of just 50ms, a request hitting 10 services in parallel has a probability of 1 − (0.99)^10 ≈ 10% of at least one being slow. With 50 downstream calls, there's a 40% chance of hitting at least one slow response. This means the user-facing p50 latency is dominated by downstream p99 latency. Mitigations: (1) hedged requests — send the same request to two servers, use the first reply. (2) Reduce fan-out. (3) Set aggressive timeouts. (4) Optimize downstream p99 aggressively. Google's Jeff Dean calls this "the tail at scale."</details>

35. What is head-of-line (HOL) blocking and how does it affect latency?

<details><summary>Answer</summary>HOL blocking occurs when a slow or stalled item at the front of a queue prevents items behind it from being processed. In HTTP/1.1: if a browser opens one TCP connection and makes 3 sequential requests, a slow response to request 1 blocks requests 2 and 3 — even if the server could handle them instantly. In TCP: a lost packet blocks delivery of all subsequent packets (even for different application streams), because TCP guarantees in-order delivery. In HTTP/2 over TCP: a single lost packet blocks all multiplexed streams. HTTP/3 (QUIC) solves transport-layer HOL blocking by making each stream independent — a loss on one stream doesn't block others.</details>

36. How does Amdahl's Law relate to throughput optimization?

<details><summary>Answer</summary>Amdahl's Law states: the maximum speedup from parallelization is limited by the serial (non-parallelizable) portion of the workload. If 20% of a request's processing is serial (e.g., acquiring a global lock, single-threaded DB write), then even with infinite parallelism, throughput can only improve by 5x (1/0.2). This means: (1) adding more workers beyond the serial bottleneck gives diminishing returns. (2) Optimizing the serial portion has outsized impact on throughput. (3) Lock contention, single-threaded writes, and global coordination are throughput ceilings. To improve throughput, you must identify and minimize serial sections — partition data to reduce lock scope, use lock-free algorithms, or shard databases to eliminate single-writer bottlenecks.</details>

37. Explain the difference between goodput, throughput, and bandwidth.

<details><summary>Answer</summary>**Bandwidth** = maximum theoretical capacity of a link (1 Gbps). **Throughput** = actual total data rate achieved on the link, including all overhead (800 Mbps). **Goodput** = the rate of **useful** application data — excluding protocol headers, retransmissions, and overhead (750 Mbps). Goodput ≤ Throughput ≤ Bandwidth. For example, sending a 1 MB file over a 1 Gbps link: bandwidth = 1 Gbps, throughput might be 800 Mbps (due to TCP/IP overhead and congestion), and goodput is the rate at which the actual file data arrives — after subtracting TCP/IP headers, retransmitted packets, and acknowledgment traffic. Goodput is what the end user actually experiences.</details>

38. How do congestion control algorithms (TCP Reno, Cubic, BBR) affect throughput?

<details><summary>Answer</summary>Congestion control determines how fast TCP sends data to avoid overwhelming the network. **TCP Reno**: uses loss-based detection — halves the window on every packet loss. Slow recovery on high-bandwidth, high-latency links. **TCP Cubic**: also loss-based but uses a cubic function for window growth, recovering faster than Reno. Default on Linux. Better for high-BDP links. **BBR** (Bottleneck Bandwidth and RTT): model-based — estimates the bottleneck bandwidth and minimum RTT, then paces packets to match. Doesn't react to loss alone. Achieves near-optimal throughput on LFNs and tolerates moderate packet loss without halving throughput. BBR can improve throughput by 2–25x over Cubic in lossy environments. Choice of algorithm significantly impacts how much of the available bandwidth is actually utilized.</details>

39. Why is it important to measure latency at the client, not just the server?

<details><summary>Answer</summary>Server-side latency only captures processing time — it misses: (1) **Network latency** — propagation and transmission delays both ways. (2) **DNS resolution** — client-side delay before the request is even sent. (3) **TLS handshake** — client perceives this; server starts measuring after the handshake. (4) **Queuing at load balancers/proxies** — time between the request leaving the client and reaching the server. (5) **Client-side rendering** — for web apps, the user experience includes browser parsing and rendering. A server logging 10ms response time might correspond to a user experiencing 500ms due to geographic distance, DNS lookup, TLS, and rendering. Always measure end-to-end, client-perceived latency (real user monitoring / RUM).</details>

40. What is the "latency-bandwidth product" and how does it affect protocol design?

<details><summary>Answer</summary>The latency-bandwidth product (same as Bandwidth-Delay Product / BDP) is `Bandwidth × RTT`, representing the "volume" of data the network pipe can hold. It profoundly affects protocol design: (1) **TCP window size** must match or exceed BDP for full utilization — which is why TCP window scaling was introduced. (2) **Request-response protocols** (like HTTP/1.1) are inherently limited because they wait for each response before sending the next request — the pipe is mostly empty. (3) **Streaming and pipelining** protocols fill the pipe better. (4) High-BDP paths need protocols that tolerate loss without drastically reducing throughput (like QUIC/BBR). Protocol designers must account for BDP to avoid leaving most of the bandwidth unused.</details>

## Scenario-Based Questions
41. Your API has 200ms average latency but users are complaining about slowness. What would you investigate?

<details><summary>Answer</summary>(1) **Check percentile latencies** — the average might hide a terrible p99. If p99 is 5 seconds, 1% of users experience unacceptable delays. (2) **Measure client-side latency** — server-measured 200ms might be 800ms for users due to network, DNS, and TLS overhead. (3) **Check geographic distribution** — users far from servers experience higher propagation delay. (4) **Investigate specific endpoints** — some endpoints might be much slower than others, dragging the average up. (5) **Check for intermittent issues** — database lock contention, GC pauses, or external dependency timeouts causing spikes. (6) **Check frontend performance** — slow rendering, large JavaScript bundles, unoptimized images adding to perceived latency.</details>

42. Your server handles 1,000 RPS but you need it to handle 10,000 RPS. What approaches would you consider?

<details><summary>Answer</summary>(1) **Reduce per-request latency** — if each request takes 10ms on a single-threaded system, throughput = 100 RPS per worker. Reducing latency to 1ms gives 1,000 RPS per worker. (2) **Add concurrency** — increase worker threads/processes (10 workers × 1,000 RPS each = 10,000 RPS). (3) **Horizontal scaling** — add more servers (10 servers × 1,000 RPS each). (4) **Caching** — cache frequent responses (Redis/Memcached) to avoid redundant computation. (5) **Async processing** — offload non-critical work to queues. (6) **Optimize database** — add read replicas, indexes, connection pooling. (7) **Batching** — combine multiple operations into one call. (8) **Protocol upgrade** — HTTP/2 multiplexing reduces connection overhead. Identify the bottleneck first, then target it.</details>

43. You have a 1 Gbps link between two data centers with 50ms RTT. The actual file transfer speed is only 10 Mbps. What could be wrong?

<details><summary>Answer</summary>The most likely culprit is **TCP window size being too small**. BDP = 1 Gbps × 50ms = 125 MB/s × 0.05s = 6.25 MB. If the TCP window is only 64 KB (default without window scaling), max throughput = 64 KB / 0.05s = 1.28 MB/s ≈ 10 Mbps. Fix: (1) enable TCP window scaling (`net.ipv4.tcp_window_scaling = 1`). (2) Increase receive buffer sizes (`net.core.rmem_max`). (3) Check for packet loss — even 0.1% loss with TCP Reno causes massive throughput degradation on high-BDP links. (4) Consider switching to BBR congestion control. (5) Use parallel TCP connections to work around per-connection window limits. (6) Verify no traffic shaper or QoS policy is throttling.</details>

44. After deploying to a new region, latency for users in that region didn't improve. What would you check?

<details><summary>Answer</summary>(1) **DNS resolution** — are DNS records actually directing users to the nearest region? Check GeoDNS or anycast configuration. (2) **CDN configuration** — is the CDN pulling from the local region or still routing to the old origin region? (3) **Backend calls** — the frontend may be local, but if it calls backend APIs in the original region, latency won't improve. Check for cross-region API calls. (4) **Database location** — if the database is still in the old region, read/write latency remains high. Need local read replicas. (5) **Health checks** — the new deployment might be failing health checks, causing traffic to fall back to the old region. (6) **Routing/BGP** — network routing might not be optimal, making packets take a longer path than expected.</details>

45. A microservice with 5ms latency is causing 500ms user-facing latency. How is this possible?

<details><summary>Answer</summary>Several possibilities: (1) **Sequential calls** — the request path calls this service 10 times sequentially: 10 × 50ms (including network) = 500ms. (2) **Tail latency amplification** — the service's p99 is much higher than 5ms, and in a fan-out architecture, the slowest call determines overall latency. (3) **Network overhead** — each call incurs serialization, network hop, deserialization (adding 20–40ms per call). (4) **Cascading calls** — this service calls another service, which calls another, creating a deep call chain. (5) **Connection establishment** — if not using connection pooling, each call pays the TCP/TLS handshake cost. (6) **Queuing** — requests waiting in thread pools or connection pools add delay. Trace the full request path end-to-end using distributed tracing (Jaeger, Zipkin).</details>

46. Your database query takes 2ms, but the API endpoint takes 200ms. Where is the extra latency?

<details><summary>Answer</summary>Investigate the full request lifecycle: (1) **Network latency** — client → load balancer → server adds propagation time. (2) **Connection pool wait** — if all DB connections are busy, the request queues for a free connection. (3) **ORM overhead** — object-relational mapping (e.g., Hibernate) adds serialization/deserialization, lazy loading, and N+1 query problems. (4) **Multiple DB calls** — the endpoint may make 20+ queries that individually are fast but add up. (5) **External API calls** — calling a third-party service (payment gateway, email) mid-request. (6) **Serialization** — converting response objects to JSON can be expensive for large payloads. (7) **Middleware** — authentication, authorization, logging, rate limiting in the pipeline. Use profiling tools (APM, flame graphs) to pinpoint where time is spent.</details>

47. During Black Friday, your throughput drops from 50,000 RPS to 10,000 RPS. What's happening?

<details><summary>Answer</summary>(1) **Resource saturation** — CPU, memory, or I/O hitting 100% causes excessive queuing and context switching, reducing effective throughput. (2) **Database bottleneck** — write contention, lock escalation, or connection pool exhaustion under spike load. (3) **Thread pool exhaustion** — all worker threads are busy waiting on slow downstream calls, so new requests are rejected. (4) **Cascading failures** — one overloaded service slows down, causing callers to timeout and retry, amplifying load. (5) **Cache stampede** — popular cache entries expire simultaneously, sending all requests to the database. (6) **GC pressure** — increased object allocation triggers frequent full GC pauses. (7) **Load balancer misconfiguration** — uneven distribution sending too much traffic to a subset of servers. Immediate actions: scale out, enable circuit breakers, shed low-priority traffic, and increase cache TTLs.</details>

48. You're designing a system for users in 10 countries. How would you think about latency, throughput, and bandwidth?

<details><summary>Answer</summary>**Latency**: Deploy in multiple regions (e.g., US, EU, Asia) to reduce propagation delay. Use CDN for static assets. Use GeoDNS or anycast to route users to the nearest region. Target < 100ms p95 latency per region. **Throughput**: Estimate per-region RPS based on user distribution. Size server fleet per region accordingly. Use auto-scaling for peak traffic. Database read replicas per region for read throughput. **Bandwidth**: Estimate per-region data transfer — video/image heavy apps need more bandwidth. Choose regions with adequate network capacity. Consider egress costs (inter-region data transfer pricing). Plan for bandwidth peaks (e.g., 3–5x normal during events). All three metrics should be planned and monitored per region, not just globally.</details>

49. A streaming service buffers frequently despite users having 100 Mbps connections. What could be wrong?

<details><summary>Answer</summary>(1) **Server-side throughput bottleneck** — the origin or CDN node can't serve video fast enough (CPU, disk I/O, or connection limits). (2) **High latency** — if RTT is 200ms, adaptive bitrate streaming reacts slowly to available bandwidth, causing buffer underruns. (3) **Packet loss** — even 1% loss on a high-bandwidth link causes TCP retransmissions and throughput collapse. (4) **Congestion at peering points** — ISP-to-CDN interconnects may be saturated, even though the last mile is 100 Mbps. (5) **Small TCP buffers** — server or client TCP buffer size limits throughput. (6) **DNS/routing issues** — connecting to a far CDN node instead of the nearest one. (7) **Adaptive bitrate misconfiguration** — requesting too high a quality for the achievable throughput. (8) **Shared bandwidth** — other devices or tabs consuming the user's 100 Mbps.</details>

50. Your system needs to transfer 1 TB of data daily between data centers. How would you reason about bandwidth and latency?

<details><summary>Answer</summary>**Bandwidth calculation**: 1 TB = 8 Tb. Over 24 hours: 8 Tb / 86,400s ≈ 93 Mbps sustained throughput required. A 1 Gbps link can easily handle this with headroom. Over 8 hours (business hours): ≈ 278 Mbps needed. **Latency consideration**: If using TCP with 50ms RTT, ensure TCP window scaling is enabled and BDP is accounted for: BDP = 1 Gbps × 0.05s = 6.25 MB. Multiple parallel streams can help fill the pipe. **Practical considerations**: (1) Use parallel transfer (multiple threads/connections). (2) Compress data before transfer to reduce volume. (3) Consider incremental sync (only changed data). (4) Budget for retransmissions and overhead (~10–20% extra capacity). (5) If time-critical, consider dedicated links or AWS Direct Connect / Azure ExpressRoute over public internet. "Never underestimate the bandwidth of a station wagon full of tapes hurtling down the highway" — Tanenbaum.</details>

## Interview-Focused Questions
51. In a system design interview, how would you explain the relationship between latency and throughput?

<details><summary>Answer</summary>"Latency and throughput are related but independent metrics. Latency measures how long a single request takes; throughput measures how many requests the system handles per second. In a single-threaded system, they're inversely related: `throughput = 1/latency`. But with concurrency, you can increase throughput without changing latency — just add more workers. The key insight is: reducing latency improves throughput (fewer resources tied up per request), but adding servers improves throughput without changing latency. For example, adding a cache reduces latency AND improves throughput. Adding more servers improves throughput but doesn't change individual request latency. In system design, I'd first identify whether the problem is latency-bound (users see slowness) or throughput-bound (system can't handle volume), because the solutions differ."</details>

52. You're asked to design a system with < 100ms latency and 100,000 RPS. How do you approach this?

<details><summary>Answer</summary>(1) **Latency budget**: 100ms total. Break it down: 20ms network, 10ms load balancer, 50ms application logic, 20ms database. Every component must fit within its budget. (2) **Throughput calculation**: Using Little's Law — L = λ × W = 100,000 × 0.1 = 10,000 concurrent requests. Need enough capacity for 10,000 in-flight requests. (3) **Architecture**: Multi-region deployment for latency. Stateless application servers behind an L7 load balancer. In-memory caching (Redis) with < 1ms reads. Database with read replicas and sharding. CDN for static assets. (4) **Sizing**: If each server handles 2,000 RPS, need 50+ servers. (5) **Monitoring**: Track p50, p95, p99 latency and RPS per server. Alert if p95 > 80ms or total RPS drops below target. (6) **Reliability**: Auto-scaling, circuit breakers, graceful degradation.</details>

53. When should you optimize for latency vs throughput?

<details><summary>Answer</summary>**Optimize for latency when**: (1) User-facing interactive applications — search, e-commerce checkout, real-time gaming, chat. Users perceive delays > 100ms. (2) Trading systems — microseconds matter for competitive advantage. (3) Time-critical APIs — payment processing, fraud detection. **Optimize for throughput when**: (1) Batch processing — ETL pipelines, log processing, ML training. Individual job latency matters less than total volume processed. (2) Message queue consumers — processing rate matters more than speed per message. (3) Data ingestion — IoT sensors, analytics events — volume over speed. **Both equally**: Social media feeds (low latency for user experience + high throughput for scale), video streaming (low buffering + high data rate). Identify the bottleneck: if users complain about speed → latency. If the system can't handle load → throughput.</details>

54. How would you diagnose whether a performance issue is latency-related, throughput-related, or bandwidth-related?

<details><summary>Answer</summary>**Latency issue**: Individual requests are slow, but the system isn't overloaded. CPU/memory utilization is low. Users see "slow load times." Diagnose: check per-request timing breakdown (DNS, TCP, TLS, server processing, DB query), distributed tracing, slow query logs. **Throughput issue**: System can't handle the request volume. Queue depths are growing. Requests are timing out or being rejected. CPU or database connections are maxed out. Diagnose: monitor RPS vs capacity, thread pool utilization, connection pool exhaustion, queue depths, error rates. **Bandwidth issue**: Large data transfers are slow. Network utilization is near 100%. Packet loss due to congestion. Diagnose: monitor network I/O (bytes in/out), interface utilization, check for throttling, measure actual vs theoretical transfer rates. Often, problems are interconnected — bandwidth saturation increases latency, which reduces throughput.</details>

55. A non-technical stakeholder asks: "We upgraded our network to 10 Gbps, why is the app still slow?" How would you explain?

<details><summary>Answer</summary>"Upgrading bandwidth (the network speed) is like adding more lanes to a highway. It helps when the road is congested — too much traffic for too few lanes. But if the app is slow because of long distances (latency) or slow toll booths (server processing), more lanes won't help. Our app might be slow because: (1) the server is far from users — adding lanes doesn't shorten the road, (2) the database queries are slow — a wider pipe doesn't make the database think faster, (3) the app has inefficient code — more bandwidth doesn't fix code that takes 2 seconds to generate a response. We should investigate where the actual delay is happening. If we're not even using 10% of our old bandwidth, upgrading to 10 Gbps won't change anything." Use monitoring data to show the actual bottleneck.</details>

56. How does geographic distribution affect each of these three metrics?

<details><summary>Answer</summary>**Latency**: Most directly affected. Every 1,000 km of distance adds ~5ms of propagation delay in fiber. US-to-Europe RTT ≈ 70–80ms. US-to-Asia RTT ≈ 150–200ms. Deploying closer to users directly reduces latency. **Throughput**: Indirectly affected. Higher latency between regions means each connection has lower throughput (more time waiting for ACKs). Database writes to a distant primary are slower, reducing write throughput. But total system throughput is improved by distributing load across regions. **Bandwidth**: Mostly unaffected by distance — bandwidth is a property of the link, not its length. However, long-distance links (undersea cables) may have limited total capacity, and they cost more. Inter-region data transfer incurs egress costs. BDP increases with distance, requiring larger TCP windows to utilize available bandwidth.</details>

57. How do caching strategies differently impact latency, throughput, and bandwidth?

<details><summary>Answer</summary>**Latency impact**: Cache hits are served in microseconds (Redis) vs milliseconds (database) or seconds (external API) — massive latency reduction for individual requests. **Throughput impact**: Caching reduces load on backends. If 80% of requests are cache hits, the database sees only 20% of traffic — effectively 5x throughput increase without adding database capacity. **Bandwidth impact**: Caching at the edge (CDN) reduces bandwidth consumption on origin server links and inter-region links. Browser caching (`Cache-Control` headers) eliminates network transfer entirely for repeated asset requests. **Cache layers**: Browser cache → CDN → Application cache (Redis) → Database query cache. Each layer reduces latency, increases effective throughput, and saves bandwidth. The closer the cache is to the user, the greater the latency and bandwidth savings.</details>

58. How would you set SLAs/SLOs for latency, throughput, and bandwidth for a production service?

<details><summary>Answer</summary>**Latency SLO**: Define in percentiles, not averages. Example: "p50 < 50ms, p95 < 200ms, p99 < 500ms." Choose based on user expectations — e-commerce checkout needs stricter SLOs than background analytics. **Throughput SLO**: "System must sustain 50,000 RPS at p99 < 500ms." Throughput SLOs always include a latency constraint — handling 100K RPS means nothing if every request takes 30 seconds. **Bandwidth SLO**: Less common for applications, more relevant for infrastructure. Example: "Video streaming: sustain 5 Mbps per concurrent viewer, support 10,000 concurrent viewers = 50 Gbps." **Error budget**: SLO compliance target (e.g., 99.9% of 5-minute windows meet latency SLO). Track compliance over 30-day rolling windows. Alert when error budget is being consumed too fast.</details>

59. Explain how a load balancer affects latency and throughput.

<details><summary>Answer</summary>**Latency**: A load balancer adds a small hop (0.1–2ms for L4, 1–10ms for L7 with SSL termination). However, it improves effective latency by: (1) routing to the fastest/closest backend, (2) terminating TLS at the edge (offloading from backends), (3) keeping persistent connections to backends (reducing per-request handshake costs). **Throughput**: Load balancers improve throughput by distributing requests across multiple servers — if each server handles 5,000 RPS and you have 20 servers, the system handles 100,000 RPS. However, the load balancer itself can become a throughput bottleneck — a single L7 load balancer might max out at 50,000–100,000 RPS. Solutions: L4 load balancing (higher throughput), multiple LB instances, or DNS-based load balancing. Algorithm choice matters: round-robin vs least-connections vs consistent hashing affects both latency and throughput uniformity.</details>

60. In a system design interview, how would you compare latency, throughput, and bandwidth using numbers for common operations?

<details><summary>Answer</summary>Key numbers every engineer should know: **Latency**: L1 cache read: 0.5ns. L2 cache: 7ns. RAM access: 100ns. SSD random read: 16µs. HDD seek: 4ms. Round trip within a datacenter: 0.5ms. San Francisco to NYC: 40ms RTT. SF to London: 80ms. SF to Singapore: 180ms. **Throughput**: Redis GET: ~100,000 ops/s per node. PostgreSQL: ~5,000–20,000 simple queries/s. Kafka: 100,000–1,000,000 messages/s per broker. Nginx: ~10,000–100,000 RPS (depending on config). **Bandwidth**: HDD sequential: 100–200 MB/s. SSD: 500 MB/s. NVMe: 3–7 GB/s. 1G Ethernet: 125 MB/s. 10G Ethernet: 1.25 GB/s. DDR4 RAM: ~25 GB/s. These numbers help you quickly estimate whether a design is feasible during interviews.</details>

61. How would you explain the impact of packet loss on each metric?

<details><summary>Answer</summary>**Latency**: Packet loss triggers retransmission. A lost packet must be detected (via timeout or duplicate ACKs, taking 1–3 RTTs) and retransmitted. A single lost packet can add 100–300ms of latency on a 50ms-RTT link. **Throughput**: TCP congestion control reacts to packet loss by halving the congestion window (TCP Reno/Cubic). Even 0.1% packet loss can reduce throughput by 50% or more on high-BDP links. This is why loss is devastating on long-distance, high-bandwidth links. The formula for TCP throughput with loss: `Throughput ≈ (MSS / RTT) × (1 / √loss_rate)`. **Bandwidth**: Packet loss doesn't reduce bandwidth (the link capacity is unchanged), but it dramatically reduces usable throughput, effectively wasting bandwidth on retransmissions. 1% loss means ~1% of bandwidth is consumed by retransmitted packets, plus the window reduction effects are far worse.</details>

62. A system needs to process 1 million events per second. Walk through how you'd think about these metrics.

<details><summary>Answer</summary>**Throughput target**: 1M events/second. **Latency constraint**: Depends on use case — if real-time analytics, need < 100ms end-to-end. If log processing, minutes are acceptable. **Architecture**: Use a distributed streaming platform (Kafka) — single broker handles ~100K–1M msgs/sec. With 10 partitions across 5 brokers, achieve 1M+ easily. **Network bandwidth**: If each event is 1 KB: 1M × 1 KB = 1 GB/s = 8 Gbps. Need 10 Gbps+ network links. If events are 10 KB: 80 Gbps — may need multiple links or compression. **Consumer throughput**: If processing takes 1ms per event, need 1,000 parallel consumers (1M ÷ 1,000/s per consumer). **Storage bandwidth**: Writing 1 GB/s to disk requires NVMe SSDs (HDD can't sustain this). **Memory**: If buffering 10 seconds of events: 10 GB RAM needed. Plan each metric — throughput sizing, latency budgets, network/disk/memory bandwidth — and identify which is the bottleneck.</details>

63. How does database indexing affect latency vs throughput?

<details><summary>Answer</summary>**Latency**: Indexes dramatically reduce read latency. A query on 10M rows: without index → full table scan (100–500ms). With B+ tree index → 3–4 page reads (< 1ms). This is a 100–500x improvement for individual queries. **Throughput**: Faster queries free up database resources (connections, CPU, I/O), allowing more queries per second — directly increasing read throughput. A database handling 1,000 QPS with 10ms queries can handle 10,000 QPS with 1ms queries (same total resource usage). **Trade-off on writes**: Every index must be updated on INSERT/UPDATE/DELETE, adding latency to write operations (1–5ms per additional index). This reduces write throughput. A table with 10 indexes is much slower to write to than a table with 2. **General rule**: Index columns used in WHERE, JOIN, ORDER BY. Avoid over-indexing — every index is a latency/throughput trade-off between reads and writes.</details>

64. How do message queues affect latency and throughput in a system?

<details><summary>Answer</summary>**Latency**: Message queues **increase** end-to-end latency because they add a hop — producer → queue → consumer. The message waits in the queue until a consumer picks it up. However, they **decrease user-perceived latency** by allowing the API to respond immediately ("your order is being processed") instead of waiting for all downstream processing. **Throughput**: Queues **dramatically improve throughput** by: (1) decoupling producers and consumers — both run at their own optimal rate. (2) buffering traffic spikes — the queue absorbs bursts, and consumers process at a steady rate. (3) enabling parallel processing — multiple consumers process messages concurrently. (4) preventing cascading failures — if a downstream service is slow, messages queue up instead of blocking the producer. **Trade-off**: You exchange synchronous low-latency for asynchronous high-throughput, accepting eventual processing of work.</details>

65. What is the "numbers every engineer should know" list and how does it relate to these metrics?

<details><summary>Answer</summary>Originally from Jeff Dean at Google, these are approximate latency/bandwidth numbers for common operations (2024-ish): **Latency**: L1 cache: 1ns, L2 cache: 4ns, Main memory: 100ns, SSD random read: 16µs, HDD seek: 4ms, Datacenter RTT: 0.5ms, CA→Netherlands RTT: 150ms. **Bandwidth/Throughput**: Main memory sequential read: 4 GB/s, SSD sequential read: 1 GB/s, HDD sequential read: 100 MB/s, 1 Gbps network: 125 MB/s. **Key takeaways**: (1) Memory is 100,000x faster than disk for random access. (2) SSD is 250x faster than HDD for random reads. (3) Sequential I/O is vastly faster than random I/O. (4) Network calls across continents are 300,000× slower than L1 cache. These numbers let you back-of-the-envelope calculate whether a design can meet its latency and throughput targets. If your design requires 10,000 random HDD reads per request, you know it won't work (10,000 × 4ms = 40 seconds).</details>

66. How would you prioritize optimizations if your system is both latency-sensitive and throughput-constrained?

<details><summary>Answer</summary>**Step 1: Profile and identify the bottleneck.** Use distributed tracing (Jaeger, Zipkin), flame graphs, and APM tools to find where time and resources are spent. **Step 2: Look for optimizations that improve both.** Caching improves both latency (faster response) and throughput (less backend work). Query optimization helps both. Connection pooling helps both. These are your highest-ROI changes. **Step 3: If forced to trade off, prioritize based on business impact.** User-facing latency usually wins — a slow experience loses customers immediately. Throughput can be addressed by scaling horizontally (adding more servers), which is expensive but doesn't require code changes. **Step 4: Architect for separation.** Use async processing to handle throughput-heavy work (background jobs) while keeping the synchronous request path optimized for latency. This avoids the trade-off by splitting the workload.</details>

67. What is "bandwidth throttling" and when is it used?

<details><summary>Answer</summary>Bandwidth throttling intentionally limits the data transfer rate for specific traffic. Use cases: (1) **ISP throttling** — ISPs may limit streaming video bandwidth during peak hours to prevent congestion. (2) **API rate limiting** — cloud providers throttle bandwidth for free-tier users (e.g., S3 throttles download speed per request). (3) **QoS (Quality of Service)** — prioritize critical traffic (VoIP, trading) by throttling less important traffic (file downloads, backups). (4) **Testing** — simulate slow networks to test app behavior (Chrome DevTools throttling). (5) **Cost control** — limit egress bandwidth to manage cloud costs. (6) **Fair usage** — prevent one tenant from consuming all shared bandwidth in a multi-tenant system. Throttling trades throughput (of throttled traffic) for fairness or priority of other traffic.</details>

68. In a distributed system, how does consistency model choice affect latency?

<details><summary>Answer</summary>**Strong consistency** (linearizability): Requires coordination between replicas — writes must be acknowledged by a majority (quorum). This adds latency equal to the slowest quorum member's response time. Cross-region strong consistency can add 100–200ms per write. Reads may also need quorum or read from leader, adding latency. **Eventual consistency**: Writes are acknowledged locally (single node), so latency is minimal — just local disk write (~1ms). Reads are served from the nearest replica. Much lower latency but you may read stale data. **Causal consistency**: Middle ground — tracks causal dependencies, which requires some coordination but less than strong consistency. Latency between strong and eventual. **Bottom line**: Stronger consistency = higher latency. This is a fundamental trade-off (CAP/PACELC). Systems like DynamoDB offer tunable consistency — choose per-request based on the use case.</details>

69. How does data compression affect latency, throughput, and bandwidth usage?

<details><summary>Answer</summary>**Bandwidth**: Compression directly reduces bandwidth usage — a 10 KB response compressed to 2 KB uses 80% less bandwidth. This is especially valuable for expensive inter-region or cloud egress links. **Latency — mixed impact**: (1) Reduces transfer time (less data over the wire → faster transmission). (2) Adds CPU time for compression on sender and decompression on receiver. Net effect depends on: if bandwidth-constrained (slow link), compression reduces latency. If CPU-constrained or on a fast LAN, compression may increase latency. **Throughput**: (1) Reduces network bottleneck → more requests can be served on the same link. (2) Increases CPU usage → may reduce throughput if CPU-bound. (3) Compression level matters: gzip -1 is fast with decent compression; gzip -9 is slow with marginally better compression. Common choices: gzip/Brotli for HTTP, LZ4/Snappy for real-time systems (fast compression, moderate ratio).</details>

70. How does consistent hashing relate to throughput and latency in distributed systems?

<details><summary>Answer</summary>**Throughput**: Consistent hashing enables horizontal scaling by distributing keys across multiple nodes. Adding a node to the ring increases total system throughput proportionally, with minimal key migration (only `k/n` keys move). Without consistent hashing, adding a node would cause massive cache invalidation, temporarily destroying throughput. **Latency**: By distributing load evenly (via virtual nodes), consistent hashing prevents hotspots — no single node is overwhelmed with disproportionate traffic, which would spike latency. However, if the hash function routes a key to a geographically distant node, latency increases. **Combined**: Consistent hashing is a core building block for achieving both high throughput (through horizontal distribution) and consistent latency (through even load distribution). Systems like Cassandra, DynamoDB, and Memcached rely on it for both metrics.</details>
