# Reliability — Question Bank

## Beginner Level
1. What is reliability in the context of system design?

<details><summary>Answer</summary>Reliability is the probability that a system will perform its intended function **correctly** and **consistently** over a given period of time, under specified conditions. It is not just about whether the system is up (that's availability), but whether the system is doing what it should. A payment system that is always running but occasionally charges customers twice is available but unreliable. Reliability means producing the right output, not just any output.</details>

2. How does reliability differ from availability?

<details><summary>Answer</summary>Availability asks **"Is the system responding?"** — a system is available if it returns an HTTP 200 when you call it. Reliability asks **"Is the response correct?"** — the balance returned is accurate, the message is delivered to the right person, the charge is the correct amount. A system can be highly available (99.99% uptime) but unreliable if it serves incorrect data. Conversely, a system that is offline for maintenance is unavailable but still considered reliable if it always produces correct results when it is running. You want both, but they are distinct properties.</details>

3. What is the difference between reliability and fault tolerance?

<details><summary>Answer</summary>Fault tolerance asks **"Does the system keep working when components fail?"** — for example, continuing to serve traffic when one database replica goes down. Reliability asks **"Is the system producing correct results?"** A fault-tolerant system can survive a node failure and keep running, but if it returns wrong data during the failover, it is fault-tolerant but not reliable. The concepts are related — fault tolerance is one tool for achieving reliability — but they are independent properties. You can optimize for one while accidentally sacrificing the other.</details>

4. What is the difference between reliability and durability?

<details><summary>Answer</summary>Durability asks **"Is data preserved despite failures?"** — for example, data survives a disk crash because it was replicated. Reliability asks **"Is the system producing correct results?"** A database that loses writes during failover is fault-tolerant (it continues operating) but not durable (data was lost). And if those lost writes cause stale data to be served, it's also not reliable. Durability is specifically about data persistence, while reliability is the broader concept of the system behaving correctly over time.</details>

5. Why is an unreliable system worse than an unavailable one from a user-trust perspective?

<details><summary>Answer</summary>When a system is unavailable, the user knows something is wrong — they see an error page, they can retry later, and they understand the system is "down." But when a system is unreliable — it's online but producing wrong results — the user may not even realize it. A payment system that silently doubles-charges, or a messaging app that delivers messages out of order, erodes trust invisibly. Users act on wrong information, make bad decisions, or get charged incorrectly. By the time the issue is discovered, the damage is already done. Unreliable systems are harder to detect, harder to debug, and destroy user confidence faster than downtime.</details>

6. What is the MTBF metric and how is it calculated?

<details><summary>Answer</summary>MTBF stands for **Mean Time Between Failures**. It measures the average time between consecutive failures. Formula: `MTBF = Total Operating Time / Number of Failures`. Example: if a system ran for 10,000 hours and experienced 5 failures, MTBF = 10,000 / 5 = 2,000 hours (roughly one failure every 83 days). MTBF helps with fleet planning — if you have 100 servers each with MTBF of 2,000 hours, you can expect approximately one server failure per day across your fleet. A higher MTBF means failures are less frequent.</details>

7. What is MTTR and why is it often more important to reduce than MTBF?

<details><summary>Answer</summary>MTTR stands for **Mean Time To Recovery**. It measures how long it takes to restore the system after a failure. Formula: `MTTR = Total Downtime / Number of Failures`. Example: 5 failures with a total repair time of 10 hours → MTTR = 2 hours per failure. MTTR includes detection time, diagnosis time, repair time, and verification time. Reducing MTTR often has more impact than reducing failure rate because: (1) failures are inevitable, especially at scale, (2) fast recovery limits the blast radius of any single failure, and (3) reducing MTTR from 4 hours to 10 minutes with automated recovery has a dramatic impact on uptime. If you cannot prevent failures, at least recover quickly.</details>

8. What is error rate and what are typical targets for different system types?

<details><summary>Answer</summary>Error rate is the percentage of requests that result in errors: `Error Rate = (Failed Requests / Total Requests) × 100%`. Typical targets: **Critical systems** (payment processing, healthcare): < 0.01% (1 in 10,000 requests fails). **Standard systems** (web applications, APIs): < 0.1% (1 in 1,000 requests fails). **Tolerant systems** (analytics, batch jobs): < 1% (1 in 100 requests fails). Error rate alone is not enough — you also need to measure data correctness, because a "successful" response can still contain wrong data.</details>

9. What is data correctness and why is it an often-overlooked reliability metric?

<details><summary>Answer</summary>Data correctness measures the percentage of responses that contain the **right** data: `Correctness = (Correct Responses / Total Responses) × 100%`. It is often overlooked because traditional monitoring focuses on availability and error rates. A system can have 99.99% availability and 0.01% error rate, but if 1% of *successful* responses contain wrong data, you have a reliability problem the standard metrics don't catch. The user received a response, it was a 200 OK, but the balance is wrong, the order total is miscalculated, or the message was corrupted. Detecting correctness issues requires data validation, checksums, reconciliation processes, and end-to-end testing — not just uptime monitoring.</details>

10. What does "under specified conditions" mean in the definition of reliability?

<details><summary>Answer</summary>"Under specified conditions" means we define what **normal operation** looks like for the system — expected load range, valid input types, supported hardware, environmental assumptions, etc. A system designed for 1,000 RPS might be reliable at that load but unreliable at 10,000 RPS. That doesn't mean it's a poor system — it means it's exceeding its specified conditions. This distinction is important because: (1) reliability targets must be tied to expected operating conditions, (2) exceeding specified conditions is a separate concern (resilience/overload handling), and (3) it allows engineers to set clear expectations with stakeholders about where the system is guaranteed to behave correctly.</details>

## Intermediate Level
11. What are the main categories of faults that make systems unreliable?

<details><summary>Answer</summary>Five main categories: (1) **Hardware failures** — disks, memory, CPUs, NICs wear out and fail. With enough hardware, failures become a daily occurrence. (2) **Software bugs** — the leading cause of reliability problems. Unlike hardware failures (random and independent), software bugs are systematic — every request hitting the buggy code path fails. Worst are bugs that silently produce wrong results. (3) **Configuration errors** — surprisingly common and often catastrophic. A typo can bring down an entire service (e.g., AWS S3 2017 outage). (4) **Human error** — operators mistype commands, engineers deploy untested changes, on-call misdiagnoses problems. Studies show this is the leading cause of outages. (5) **Overload and cascading failures** — when one component slows down, requests queue, timeouts fire, retries multiply load, and the problem cascades.</details>

12. Why are software bugs more dangerous to reliability than hardware failures?

<details><summary>Answer</summary>Hardware failures are well-understood and follow predictable statistical patterns — they're random and independent (one disk failing doesn't cause the next disk to fail). This means you can model and plan for them with redundancy. Software bugs, however, are **systematic** — every single request that hits the buggy code path fails. If one server has a bug, every server running the same code has the same bug. The most dangerous software bugs are those that don't crash the system but **silently produce wrong results**. A crash gets detected immediately; a subtle data corruption might go unnoticed for weeks. Hardware failures degrade capacity; software bugs degrade correctness.</details>

13. How do configuration errors cause reliability problems?

<details><summary>Answer</summary>Configuration mistakes are surprisingly common and often catastrophic for several reasons: (1) **Blast radius is huge** — a single typo in a configuration file can bring down an entire service, as opposed to a bug that might affect only one code path. (2) **They bypass testing** — code is unit tested and reviewed, but configuration changes often lack the same rigor. (3) **They're deployed immediately** — many configuration systems apply changes in real-time without canary testing. (4) **They're hard to roll back** — unlike code deployments, config rollbacks may not be automated. The AWS S3 outage in 2017 was triggered by a command intended to remove a small number of servers that accidentally removed a much larger set. This single configuration mistake caused a major internet-wide disruption.</details>

14. What is a cascading failure and how does it relate to reliability?

<details><summary>Answer</summary>A cascading failure occurs when the failure of one component triggers failures in dependent components, which trigger more failures, creating a domino effect. The typical chain: (1) One component slows down under load. (2) Callers' requests queue up, waiting for the slow component. (3) Callers hit timeouts, which are often set too high (consuming resources while waiting). (4) Callers retry failed requests, **multiplying** the load on the already-struggling component. (5) The overload spreads upstream — now the callers themselves become slow, and *their* callers start timing out and retrying. Systems that work perfectly under normal load can fail catastrophically this way. Cascading failures are especially dangerous because they can take down an entire distributed system from a single point of stress.</details>

15. What is redundancy and how does it improve reliability?

<details><summary>Answer</summary>Redundancy means having **backup components** ready to take over if one part fails. Types include: (1) **Server redundancy** — multiple application servers behind a load balancer; if one crashes, others continue. (2) **Network redundancy** — duplicate network paths so that a single cable cut doesn't take the service offline. (3) **Database redundancy** — replicas so a primary failure doesn't mean data loss. (4) **Geographic redundancy** — deploying across data centers or regions so a localized disaster doesn't cause total failure. Redundancy is the most fundamental reliability technique because it decouples the reliability of the system from the reliability of any single component. However, it adds cost and complexity (keeping replicas consistent, handling failover correctly).</details>

16. What is failover and what are the different types?

<details><summary>Answer</summary>Failover is the process by which a system **automatically switches** to a redundant or standby component when a failure is detected. Types: (1) **Active-passive (cold/warm failover)** — a standby replica takes over when the active node fails. Simple but has a brief downtime during switchover. (2) **Active-active (hot failover)** — multiple nodes serve traffic simultaneously. If one fails, the others absorb the load. Near-zero downtime but more complex (data consistency, split-brain risk). (3) **Database failover** — promoting a replica to primary when the primary fails. Risks include data loss if replication was asynchronous. Failover ensures continuous operation but must be tested regularly — many outages happen because the failover itself was broken.</details>

17. What is graceful degradation and why is it important for reliability?

<details><summary>Answer</summary>Graceful degradation means that when parts of the system fail, **core functionality still works** rather than the system going completely offline. It's a spectrum of service levels: **Full service** — all features available (personalized recommendations, real-time inventory, all payment options). **Partial service** — non-critical features disabled (generic recommendations, cached inventory). **Core only** — essential functions only (browse products, basic checkout). **Emergency mode** — minimal functionality (cached pages, orders queued for later). This is important because: (1) total failure is rarely acceptable — a partially working e-commerce site is better than a 503 error. (2) It prioritizes business-critical paths. (3) It gives engineering time to fix the issue without a total outage. Well-designed degradation requires identifying which features are critical vs. optional upfront.</details>

18. How does a circuit breaker pattern work and why is it used?

<details><summary>Answer</summary>A circuit breaker prevents cascading failures in microservices by detecting when a dependency is failing and temporarily cutting off requests. It has three states: **Closed** — requests pass through normally; failures are counted. **Open** — requests fail immediately *without* calling the dependency (fast-fail). This prevents wasting resources and gives the dependency time to recover. **Half-open** — after a timeout, a limited number of test requests are allowed through. If they succeed, the circuit closes (back to normal). If they fail, it reopens. Transitions: Closed → Open when failures exceed a threshold (e.g., 5 failures in 30 seconds). Open → Half-open after a timeout (e.g., 30 seconds). Half-open → Closed on success, or back to Open on failure. Without circuit breakers, a failing dependency causes all callers to hang, exhaust thread pools, and cascade the failure upstream.</details>

19. What is idempotency and why is it critical for reliability?

<details><summary>Answer</summary>An operation is **idempotent** if executing it multiple times produces the same result as executing it once. This is critical for reliability because network failures make it unclear whether a request succeeded or failed — the client sends a request, the server processes it, but the response is lost. If the client retries, a non-idempotent operation (like "transfer $100") would execute twice, resulting in $200 transferred. Idempotency solves this with a unique key (e.g., `ID: abc123`). The server stores the key on first execution. Subsequent retries with the same key return the stored result without re-executing. Stripe, PayPal, and other payment processors **require** idempotency keys for money-moving operations. Without them, network issues could cause duplicate charges.</details>

20. How does load balancing contribute to system reliability?

<details><summary>Answer</summary>Load balancing improves reliability in several ways: (1) **Eliminates single points of failure** — traffic is distributed across multiple servers, so one server crashing doesn't bring down the service. (2) **Prevents overload** — even distribution ensures no single server is overwhelmed, which prevents latency spikes and cascading failures. (3) **Health checking** — load balancers continuously check server health and stop sending traffic to unhealthy nodes. (4) **Graceful handling of failures** — when a server fails its health check, it's automatically removed from the pool, and traffic is rerouted to healthy servers without user impact. However, the load balancer itself can be a single point of failure — so production systems use redundant load balancers (active-passive or active-active pairs).</details>

21. What role does monitoring and alerting play in reliability?

<details><summary>Answer</summary>Monitoring and alerting are essential because they directly reduce **MTTR** (Mean Time To Recovery). A reliable system is constantly monitored through: (1) **Health metrics** — CPU, memory, disk, network utilization. (2) **Application metrics** — error rates, latency percentiles, throughput. (3) **Business metrics** — transaction success rates, data correctness checks. (4) **Alerting** — notifying engineers of issues *before* they escalate. An alert on rising error rates at 0.5% can prevent a full outage at 5%. (5) **Dashboards** — real-time visibility into system health. Without monitoring, you only discover failures when users complain — by then, MTTR includes user-reporting time, which can be hours. With proper alerting, you detect issues in seconds. The goal is always to know about problems before your users do.</details>

22. How can a system be both highly available and unreliable simultaneously?

<details><summary>Answer</summary>A system is available if it responds to requests, and reliable if it responds **correctly**. These can diverge: (1) **Aggressive caching** — a system aggressively caches data for availability but serves stale (incorrect) responses. The cache returns fast but the data is outdated — available but unreliable. (2) **Payment system** — always processes transactions (available) but occasionally charges customers twice (unreliable). (3) **Messaging app** — always responds (available) but delivers messages out of order (unreliable). (4) **Database with async replication** — reads from replicas are always available but may return data that hasn't caught up with the primary (stale reads). In each case, optimizing for one property (availability/speed) accidentally sacrifices another (reliability/correctness).</details>

## Advanced Level
23. How do you design an idempotency system for a payment service?

<details><summary>Answer</summary>Key components: (1) **Client generates an idempotency key** — a UUID (e.g., `abc123`) sent with every request. This key is unique per intended operation. (2) **Server stores the key + result** — on first execution, the server stores `{key: abc123, status: completed, result: {amount: $100, transfer_id: xyz}}` atomically alongside the operation. (3) **On retry**, the server checks if the key exists. If it does, it returns the stored result *without re-executing*. (4) **Atomicity is critical** — the operation and the key storage must be in the same transaction. If they're not atomic, you risk a race condition where the operation executes but the key isn't stored, or vice versa. (5) **Key expiry** — keys should have a TTL (e.g., 24 hours) to avoid unbounded storage growth. (6) **Concurrent requests** — use database locking or compare-and-swap to handle two identical requests arriving simultaneously. The first acquires the lock, the second waits and then returns the stored result.</details>

24. Compare active-passive vs active-active failover in terms of reliability trade-offs.

<details><summary>Answer</summary>**Active-passive**: One node serves traffic (active), another waits on standby (passive). On failure, the passive takes over. **Pros**: simpler data consistency (only one writer), lower cost (passive may be smaller). **Cons**: brief downtime during switchover (seconds to minutes), passive resources are "wasted" during normal operation, failover may not work if it's never tested (especially dangerous). **Active-active**: All nodes serve traffic simultaneously. On failure, remaining nodes absorb the load. **Pros**: near-zero downtime on failure, better resource utilization, higher throughput. **Cons**: much more complex — data consistency across nodes is hard (conflict resolution, split-brain prevention), requires synchronization or conflict-free data structures. **Reliability trade-off**: Active-active is more available during failures but risks data consistency issues, which can make it less reliable. Active-passive is simpler and more reliable in terms of correctness, but has a brief availability gap during failover.</details>

25. How do you implement graceful degradation in practice? Walk through the steps.

<details><summary>Answer</summary>Step-by-step: (1) **Classify features by criticality** — categorize every feature as critical (checkout, login), important (recommendations, real-time inventory), nice-to-have (analytics, personalization). (2) **Define degradation levels** — map system health to feature sets. Level 1 (full): all features. Level 2 (partial): non-critical features disabled. Level 3 (core): essential only. Level 4 (emergency): minimal functionality. (3) **Implement feature flags** — each non-critical feature should be toggleable at runtime without deployment. (4) **Add fallback paths** — if the recommendation engine is down, show "popular items" from a cache instead of nothing. If real-time inventory is unavailable, show cached inventory. (5) **Automate transitions** — use health signals to trigger degradation automatically. If error rate exceeds 5%, disable non-critical features. (6) **Test degradation paths** — regularly simulate failures in production to verify degradation works (chaos engineering). Untested degradation paths often have bugs that cause total failure instead of graceful reduction.</details>

26. What is the relationship between retries, timeouts, and idempotency in building reliable systems?

<details><summary>Answer</summary>These three concepts form a reliability triad: **Timeouts** prevent indefinite waiting — if a dependency doesn't respond in X ms, give up and either return an error or try a fallback. Without timeouts, one slow dependency consumes all your threads/connections. **Retries** handle transient failures — network blips, momentary overload, etc. If a request fails, trying again may succeed. But retries without strategy amplify load on already-struggling systems (retry storms). Best practices: exponential backoff, jitter, and retry budgets. **Idempotency** makes retries safe — without it, retrying a "transfer $100" request might charge $200. With an idempotency key, the server detects the retry and returns the original result. Together: set reasonable timeouts → retry with exponential backoff on failure → use idempotency keys so retries are safe to execute. Remove any one of these, and the system has a reliability gap.</details>

27. How does data replication improve reliability, and what risks does it introduce?

<details><summary>Answer</summary>**Benefits**: (1) If one database node fails, replicas still have the data — improved durability and availability. (2) Read replicas reduce load on the primary, reducing the chance of overload-induced failures. (3) Geographic replicas bring data closer to users, reducing latency and providing disaster recovery. **Risks**: (1) **Replication lag** — asynchronous replication means replicas may be seconds behind the primary. A read from a replica right after a write to the primary returns stale data. This is a reliability problem. (2) **Failover data loss** — if the primary fails before async replicas catch up, committed writes are lost. (3) **Split-brain** — if network partitions isolate replicas, both might accept writes, leading to conflicting data. Resolving conflicts is hard and error-prone. (4) **Consistency vs. availability trade-off** — synchronous replication ensures consistency but increases write latency and fails if any replica is down. Choosing the right replication strategy is a key reliability decision.</details>

28. How does chaos engineering relate to reliability?

<details><summary>Answer</summary>Chaos engineering is the practice of intentionally injecting failures into production systems to discover weaknesses before they cause real outages. Pioneered by Netflix (Chaos Monkey), it's based on the principle that you can't be confident in your system's reliability unless you've tested it under failure conditions. Techniques: (1) **Kill random instances** — verifies failover and redundancy work. (2) **Inject network latency/partition** — tests timeout and retry logic. (3) **Corrupt data** — tests data validation and correctness checks. (4) **Simulate dependency failures** — tests circuit breakers and fallback paths. (5) **Resource exhaustion** — fills disk, exhausts memory, maxes CPU. Why it matters: many systems have untested failure paths. A circuit breaker that was never triggered in production might have a bug. A failover that was designed but never tested might fail when needed. Chaos engineering converts unknown unknowns into known, fixed issues.</details>

29. What is the trade-off between reliability and performance?

<details><summary>Answer</summary>Improving reliability often costs performance: (1) **Synchronous replication** — ensures no data loss (reliability) but adds write latency (performance). (2) **Strong consistency** — guarantees correct reads (reliability) but requires coordination between nodes (latency). (3) **Checksums and validation** — verify data correctness (reliability) but add CPU overhead (throughput). (4) **Idempotency checks** — prevent duplicate operations (reliability) but require database lookups (latency). (5) **Circuit breakers** — prevent cascading failures (reliability) but reject requests that might have succeeded (availability). (6) **Redundancy** — ensures failover (reliability) but costs more infrastructure (cost/complexity). The art of system design is finding the right balance. Use synchronous replication for financial transactions (reliability is paramount) but async replication for social media feeds (eventual consistency is acceptable). The trade-off should be driven by business requirements.</details>

30. How do you prevent and handle retry storms?

<details><summary>Answer</summary>A retry storm occurs when many clients retry failed requests simultaneously, overwhelming an already-struggling system. Prevention: (1) **Exponential backoff** — each retry waits exponentially longer (1s, 2s, 4s, 8s...) instead of immediately retrying. (2) **Jitter** — add randomness to backoff intervals so all clients don't retry at the exact same time. Without jitter, exponential backoff still creates synchronized retry waves. (3) **Retry budgets** — limit the total number of retries as a percentage of normal traffic (e.g., retries can add at most 10% extra load). (4) **Circuit breakers** — stop retrying entirely when the dependency is clearly down. (5) **Client-side rate limiting** — each client limits its own retry rate. (6) **Server-side load shedding** — when overloaded, reject requests with `503 + Retry-After` header, explicitly telling clients when to retry. (7) **Distinguish retriable vs. non-retriable errors** — don't retry 400s (client errors), only retry 503s (server overload).</details>

## Scenario-Based Questions
31. A payment system occasionally charges customers twice. How would you diagnose and fix this?

<details><summary>Answer</summary>**Diagnosis**: (1) Check if retry logic is sending duplicate requests — look for the same payment submitted twice within a short window. (2) Check if the server processes the request but the client doesn't receive the response (response lost, client retries). (3) Check for race conditions — two threads processing the same payment concurrently. (4) Review database transaction isolation — could the same payment ID be inserted twice? **Fix**: (1) **Implement idempotency keys** — every payment request includes a unique key. Server stores the key on first processing. Retries with the same key return the stored result without re-charging. (2) Use database unique constraints on the payment/idempotency key to prevent duplicate inserts at the database level. (3) Process payments within a database transaction that checks and inserts the idempotency key atomically. (4) Return clear response codes so the client knows whether the payment succeeded, failed, or is unknown (and should retry with the same key).</details>

32. Your system serves stale data after a database failover. What happened and how do you prevent it?

<details><summary>Answer</summary>**What happened**: The primary database failed, and a replica was promoted to primary. But the replica was using **asynchronous replication** and was behind —  some committed writes hadn't been replicated yet. After promotion, reads from the new primary return data that's missing the most recent writes. **Prevention strategies**: (1) **Synchronous replication** — ensures replicas have all committed writes before the primary acknowledges the write. No data loss on failover, but higher write latency. (2) **Semi-synchronous replication** — at least one replica must confirm the write (compromise between sync and async). (3) **Read-your-writes consistency** — after a write, route the user's subsequent reads to the primary (or a confirmed replica), not an arbitrary replica. (4) **Replication lag monitoring** — alert when replicas fall too far behind. (5) **Wait for replica catch-up before promotion** — during planned failover, ensure the replica is fully caught up before promoting. (6) **Application-level reconciliation** — after failover, detect and repair any data inconsistencies.</details>

33. Your monitoring shows 99.99% availability but users are reporting incorrect data. What do you investigate?

<details><summary>Answer</summary>This is a classic case where availability and reliability have diverged. The system is up (99.99% available) but not **correct**. Investigation: (1) **Data correctness checks** — compare responses to a known source of truth. Are balances, prices, or inventory counts wrong? (2) **Replication lag** — if reads go to replicas, check if they're behind the primary. Users may see stale data. (3) **Cache staleness** — if aggressive caching is enabled, check if caches are returning outdated data. (4) **Race conditions** — concurrent writes might be overwriting each other (lost updates). (5) **Bug in business logic** — a code change might be calculating values incorrectly for certain edge cases. (6) **Data pipeline issues** — if denormalized data is populated by a background job, check if that job is running and syncing correctly. **Action**: Add data correctness monitoring (compare random responses to source of truth), implement checksums, and create reconciliation jobs that detect and alert on data drift.</details>

34. After deploying a new version, error rates increase from 0.01% to 2%. How do you handle this?

<details><summary>Answer</summary>Immediate response: (1) **Roll back** — if you have automated rollback, trigger it immediately. Every minute at 2% error rates affects users. (2) **If rollback isn't possible**, use **feature flags** to disable the new functionality without a full rollback. (3) **Investigate the errors** — look at error logs, stack traces, and which endpoints are affected. Is it all traffic or specific request patterns? (4) **Check if it's a canary issue** — if you used canary deployments, why wasn't this caught? Was the canary too small or too brief? **Root cause analysis**: (5) Check if the new code path has a bug triggered by specific inputs. (6) Check for configuration differences between staging and production. (7) Check for data-dependent bugs (production has data patterns that staging doesn't). **Prevention**: (8) Improve test coverage for the specific failure mode. (9) Implement canary deployments with automatic rollback if error rate exceeds a threshold (e.g., 0.1%). (10) Add integration tests with production-like data.</details>

35. A single misconfigured server is returning wrong results but not throwing errors. How do you detect and prevent this?

<details><summary>Answer</summary>**Detection is hard** — the server returns 200 OK with wrong data, so standard error-rate monitoring doesn't catch it. Strategies: (1) **Data validation at the load balancer or gateway** — sanity-check response formats and key fields. (2) **Canary comparison** — periodically send the same request to multiple servers and compare responses. If one differs, flag it. (3) **Checksum/hash comparison** — compute checksums on critical data fields and compare across servers. (4) **Synthetic monitoring** — send known requests with known expected responses and validate correctness. (5) **Anomaly detection** — monitor per-server metrics (response time, response size distributions). A misconfigured server may show subtle differences. **Prevention**: (6) **Configuration management** — use tools like Ansible, Terraform, or Kubernetes ConfigMaps to ensure all servers have identical configurations. Never manually edit a single server's config. (7) **Immutable deployments** — deploy identical container images everywhere. (8) **Configuration validation** — validate configs against a schema before deployment.</details>

36. Your microservices system experiences a cascading failure. Walk through how you'd investigate and recover.

<details><summary>Answer</summary>**Immediate investigation**: (1) **Find the root cause service** — check dashboards for the first service that showed elevated latency or errors. Cascading failures propagate from one origin. (2) **Check resource utilization** — is the root service CPU-bound, OOM, or blocked on a dependency (database, external API)? (3) **Check for retry storms** — are upstream services retrying aggressively, multiplying load? **Recovery**: (4) **Shed load** — enable rate limiting or reject non-critical traffic on the root service. (5) **Open circuit breakers** — manually trip circuit breakers for upstream services to stop piling on. (6) **Scale the stressed service** — add instances if it's a capacity issue. (7) **Fix the root cause** — if it's a slow query, bad deploy, or configuration error, fix it. (8) **Gradually restore** — don't close all circuit breakers at once. Slowly ramp traffic back to prevent re-triggering. **Prevention**: configure appropriate timeouts, circuit breakers, retry limits with exponential backoff + jitter, and load shedding thresholds for all services.</details>

37. You need to transfer money between two banks. Network failures are possible. How do you ensure reliability?

<details><summary>Answer</summary>This is a classic distributed transaction problem. Approach: (1) **Idempotency keys** — every transfer request carries a unique ID. Both banks check the ID before processing to prevent duplicates. (2) **Two-phase approach** — first, debit the source bank (mark funds as "held/pending"). Then, credit the destination bank. If the credit fails, release the hold. (3) **Saga pattern** — each step has a compensating action. If step 2 (credit) fails, execute the compensating action for step 1 (reverse the debit). (4) **Persist state** — store the transfer state (initiated → debited → credited → completed) in a durable log. On crash recovery, resume from the last persisted state. (5) **Reconciliation** — run periodic reconciliation between both banks to detect any discrepancies. (6) **Timeouts with explicit resolution** — if no response is received, the transfer is in an "unknown" state. A background job checks the actual status and resolves it (complete or compensate). Never leave transfers in an ambiguous state.</details>

38. Your system works perfectly under normal load but fails under 3x spike traffic. What design changes would you make?

<details><summary>Answer</summary>(1) **Auto-scaling** — configure infrastructure to automatically add instances when load crosses thresholds. Scale based on CPU, connection count, or queue depth. (2) **Load shedding** — when overloaded, reject low-priority requests early (return 503 immediately) rather than trying to process everything and failing at everything. (3) **Graceful degradation** — disable non-critical features under high load (e.g., skip recommendations, serve cached results). (4) **Rate limiting** — limit requests per user/client to prevent any single source from consuming disproportionate resources. (5) **Queue-based buffering** — accept requests into a queue and process at a sustainable rate, smoothing out spikes. (6) **Circuit breakers** — prevent downstream overload from cascading. (7) **Pre-provisioning for predicted spikes** — if you know Black Friday is coming, scale up in advance. (8) **Load testing** — regularly test at 3x-5x expected load to identify breaking points before they hit production.</details>

## Interview-Focused Questions
39. In a system design interview, how would you explain the difference between availability and reliability?

<details><summary>Answer</summary>"Availability asks 'Is the system up?' — does it respond to requests? Reliability asks 'Is it doing what it should?' — are the responses correct? They're related but independent. A payment system that's always online but occasionally double-charges customers is highly available but unreliable. A database with 99.999% uptime but 1% stale reads is available but not reliable. From a business perspective, unreliable systems are often worse than unavailable ones — when a system is down, users know to retry. When it silently gives wrong answers, users act on bad information. In my designs, I prioritize reliability alongside availability by implementing idempotency for critical operations, data validation, reconciliation processes, and monitoring that checks not just 'are we responding?' but 'are we responding correctly?'"</details>

40. How would you design a reliable payment processing system?

<details><summary>Answer</summary>"I'd focus on five key reliability principles: (1) **Idempotency** — every payment request carries a unique idempotency key. The server stores the key and result atomically on first execution. Retries return the stored result without re-processing. This prevents duplicate charges on network failures. (2) **Exactly-once semantics** — use database transactions to ensure the payment and its record are committed atomically. (3) **State machine** — model each payment as a state machine (initiated → authorized → captured → completed/failed). Every transition is logged. On crash recovery, resume from the last known state. (4) **Reconciliation** — run hourly reconciliation between the application database, the payment gateway, and the bank ledger. Flag any discrepancies for manual review. (5) **Circuit breakers** — if the payment gateway is failing, stop sending requests (fail fast) rather than queueing them up and potentially processing them late when the gateway recovers, possibly causing duplicates. (6) **Monitoring** — track double-charge rate, refund rate, and payment success rate as core reliability metrics."</details>

41. How would you measure and improve the reliability of a distributed system?

<details><summary>Answer</summary>"I'd measure reliability with four metrics: (1) **MTBF** — how often do correctness failures occur? (2) **MTTR** — how quickly do we detect and recover? (3) **Error rate** — what percentage of requests fail? (4) **Data correctness** — what percentage of successful responses are actually correct? To improve reliability: (1) **Reduce MTTR** — invest in monitoring, alerting, and automated recovery. Detect issues in seconds, not hours. (2) **Chaos engineering** — intentionally inject failures to find weaknesses (kill instances, add latency, partition networks). (3) **Idempotency** — make all state-changing operations safe to retry. (4) **Circuit breakers** — prevent cascading failures. (5) **Graceful degradation** — define degradation levels so partial failures don't become total outages. (6) **Human error mitigation** — canary deployments, automated rollbacks, config validation, code review. The biggest reliability gains usually come from reducing MTTR and preventing human error, not from preventing hardware failures."</details>

42. An interviewer asks: "Your system is 99.99% available. Is that reliable enough?" How do you respond?

<details><summary>Answer</summary>"99.99% availability means about 52 minutes of downtime per year — which sounds great. But availability alone doesn't tell me if the system is reliable. I'd ask three follow-up questions: (1) **What's the data correctness rate?** If 1% of successful responses contain wrong data, that's a reliability problem no availability metric captures. (2) **What's the error rate during those 'available' periods?** The system might be responding, but responding with errors or degraded results. (3) **What's the MTBF and MTTR pattern?** 52 minutes could be one long outage or 52 one-minute blips — very different reliability profiles. For a payment system, I'd want not just 99.99% availability but also < 0.001% duplicate transactions, < 0.01% incorrect amounts, and an MTTR under 5 minutes. Reliability encompasses correctness, not just uptime."</details>

43. How do you handle the trade-off between strong consistency and reliability in a distributed database?

<details><summary>Answer</summary>"Strong consistency (every read returns the most recent write) improves reliability in terms of data correctness — users always see the right data. But it hurts reliability in terms of availability — if a quorum of nodes can't be reached, the system refuses to serve requests rather than serve stale data. The right choice depends on the use case: **Banking/payments** — strong consistency is non-negotiable. A user checking their balance must see the latest state. The reliability cost of showing a stale balance ($1000 when it's really $5) far exceeds the cost of a brief unavailability during a partition. **Social media feed** — eventual consistency is fine. Showing a post 5 seconds late is acceptable; refusing to show the feed entirely is not. **Shopping cart** — eventual consistency with conflict resolution. Merge conflicting carts rather than losing items (Amazon's Dynamo approach). I'd design the system with tunable consistency — use strong consistency for critical paths and eventual consistency for non-critical ones."</details>

44. Walk through how you'd design a circuit breaker implementation.

<details><summary>Answer</summary>"Implementation details: (1) **State storage** — track failure count, last failure time, and circuit state per downstream dependency. Use an in-memory data structure (thread-safe). (2) **Closed state** — all requests pass through. Maintain a sliding window of recent requests (e.g., last 30 seconds). Track success/failure counts. (3) **Trip threshold** — open the circuit if failure rate exceeds a threshold (e.g., >50% failure rate over 30 seconds, minimum 10 requests). (4) **Open state** — immediately return a fallback response or error without calling the dependency. Start a timer for the recovery timeout (e.g., 30 seconds). (5) **Half-open state** — after the timeout, allow a limited number of test requests (e.g., 3). If all succeed, close the circuit. If any fail, reopen. (6) **Fallback logic** — when the circuit is open, return cached data, a default value, or a graceful error. (7) **Monitoring** — emit metrics on circuit state changes, so you can see when dependencies are flapping. (8) **Per-operation granularity** — have separate circuit breakers per dependency endpoint, not one per service."</details>

45. How would you explain the concept of "error budgets" in the context of reliability?

<details><summary>Answer</summary>"An error budget is the acceptable amount of unreliability in a given time period, derived from the SLO. If your SLO is 99.9% reliability, your error budget is 0.1% — roughly 43 minutes of downtime or equivalent error volume per month. How it works: (1) **Track consumption** — continuously measure how much of the error budget has been used. If you've had 20 minutes of issues, you've used ~47% of the monthly budget. (2) **Drive decisions** — when the error budget is healthy (barely consumed), teams can deploy faster, run experiments, and take risks. When the budget is nearly exhausted, teams slow down: freeze deployments, focus on stability, fix reliability debt. (3) **Alignment** — product teams want to ship features fast; reliability teams want stability. Error budgets align both: 'You can move as fast as you want, as long as you don't exceed the budget.' (4) **Accountability** — if a team burns through their error budget, they must invest in reliability before shipping new features. This framework (from Google's SRE book) balances innovation with reliability."</details>

46. How do you prevent human error from causing reliability incidents?

<details><summary>Answer</summary>"Human error is the leading cause of outages. Mitigation strategies at multiple levels: **Prevention**: (1) **Canary deployments** — roll out changes to 1% of traffic first; automated metrics gate prevents full rollout if error rate increases. (2) **Config validation** — validate all configuration changes against a schema before applying. The AWS S3 outage would have been prevented by a validation check. (3) **Infrastructure as Code** — Terraform, Kubernetes manifests — changes are code-reviewed and version-controlled, not typed live. (4) **Automation** — replace manual runbooks with automated scripts. Humans mistype commands; scripts don't. **Detection**: (5) **Automated rollback** — if error rate exceeds a threshold post-deployment, automatically roll back without human intervention. (6) **Real-time alerting** — detect the impact of mistakes in seconds, not hours. **Mitigation**: (7) **Blast radius limitation** — deploy to one region at a time, one cluster at a time. A mistake affects 10% of users, not 100%. (8) **Blameless post-mortems** — analyze incidents without blame so engineers report issues freely, and systemic fixes are implemented."</details>

47. In a system design interview, how would you incorporate reliability into a URL shortener design?

<details><summary>Answer</summary>"Even for a 'simple' system like a URL shortener, reliability matters: (1) **Data durability** — the mapping between short URL and long URL must be durable. Losing this mapping means broken links forever. Use replicated database (e.g., PostgreSQL with synchronous replicas or Cassandra with RF=3). (2) **Idempotency on creation** — if a client creates a short URL and the response is lost, retrying shouldn't create a duplicate. Use the original URL as a natural idempotency key — same long URL always maps to the same short URL. (3) **Redirect correctness** — the most critical reliability requirement: every short URL must redirect to the correct long URL, every time. Cache the mapping aggressively (Redis), but ensure cache invalidation is correct. (4) **Graceful degradation** — if the database is down, serve redirects from cache. Analytics (click tracking) can degrade, but redirects shouldn't. (5) **Monitoring** — track redirect success rate (should be ~100%), error rates on creation, and cache hit rates. A 'simple' system can still have subtle reliability issues."</details>
